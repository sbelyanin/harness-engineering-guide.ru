# incident-postmortem — blameless разбор инцидентов Harness

Превращает свежий инцидент в структурированный документ: что случилось, почему, как восстановились, что изменить в процессах. Главная ценность — не «отчитаться», а **накопить institutional knowledge** и не наступить на те же грабли.

## Когда использовать

- **Любой SEV-1 / SEV-2 инцидент** — постмортем обязателен, в течение 48 часов
- **SEV-3 с неочевидной root cause** — полезно для RCA даже без дедлайна
- **Безопасность** (prompt injection detected, data leakage) — всегда
- **Cost spike > 3× от baseline** — почему не сработали alarms раньше
- **«Необычное поведение модели»** — оффлайн postmortem для накопления паттернов

## Когда НЕ использовать

- Обычные 5xx без user-impact (SLO не нарушен)
- SEV-4 (cosmetic / edge case) — backlog item, не постмортем
- Повторение уже разобранного инцидента без новых деталей — обновите старый постмортем

## Принципы blameless postmortem

Постмортем ищет **системные причины**, а не виноватых. Без этого культура превращается в «кто наступил на грабли», и люди начинают скрывать инциденты.

| ✗ Не так | ✓ Так |
|---------|-------|
| «Разработчик X залил баг» | «CI не покрыл кейс, review не нашёл, deploy без feature flag» |
| «Дежурный не отреагировал» | «Alert ушёл в мусорный канал, ack-time не отслеживался» |
| «Модель галлюцинировала» | «Guardrail rule устарел, prompt template изменился без eval regression» |
| «Кто-то не следовал процессу» | «Процесс невозможно выполнить без ручных шагов, которые забываются» |

**Правило:** если в постмортеме встречается имя человека как «причина» — переписывайте. Люди не ошибаются «от балды» — есть системная причина, почему ошибку можно было совершить и почему её не поймали раньше.

## Структура постмортема (полный template)

````markdown
# Postmortem: <краткий заголовок инцидента>

**Date:** YYYY-MM-DD  
**Severity:** SEV-X  
**Incident duration:** Xh Ym  
**Lead:** <name/role>  
**Status:** Final / Draft  
**Incident ID:** <auto-generated>

## Summary

<1-2 параграфа для менеджера/stakeholder'а: что произошло, какой impact,
сколько длилось. Без технических деталей — что случилось на человеческом
языке. Кто затронут, что не работало.>

## Impact

- **Пользователей затронуто:** ~N (или % от DAU)
- **Запросов упало:** ~M
- **SLI нарушен:** <success rate / latency / eval pass rate>
- **Error budget:** сожжено X% (см. [Alerting и SLO](/guide/alerting-and-slo))
- **Cost impact:** $X (если применимо: cost spike, лишний inference)
- **External communication:** твит/status page/internal announcement

## Timeline (UTC)

Все времена в UTC, чтобы не путаться. С точностью до минуты.

- **HH:MM** — Первые признаки (метрика отклонилась от baseline)
- **HH:MM** — Alert triggered: `<alert name>`, ack в <Xm
- **HH:MM** — Diagnosis: <что думали сначала>
- **HH:MM** — Mitigation попытка 1: <action>, результат
- **HH:MM** — Mitigation попытка 2: <action>, результат
- **HH:MM** — Recovery confirmed: метрики вернулись в норму
- **HH:MM** — Post-incident review scheduled

## Root Cause

<Техническое объяснение: что именно сломалось и почему. Цепочка «как мы сюда
попали». Если несколько факторов — нумеруйте.>

Примеры:
1. CI не покрывал edge case с пустым tool response
2. Модель YandexGPT обновилась 2026-07-XX и начала отказываться от промптов
   в формате `<instruction>` — наш prompt template стал отказоустойчивым
3. Provider rate-limit упал до 10 RPM без уведомления, failover не сработал
   из-за неверной метрики здоровья

## What Went Well

- <Что сработало, как задумано>
- <Какая автоматизация спасла>
- <Traces дали быстрый diagnosis>
- <Communication был быстрым>

## What Went Poorly

- <Что задержало recovery>
- <Какие инструменты подвели>
- <Alert был слишком поздним / слишком ранним>
- <Runbook отсутствовал / устарел>

## Action Items

Конкретные, с владельцем и сроком. Без владельца = не будет сделано.

- [ ] **(Owner: @name, due: YYYY-MM-DD)** Fix <root cause technical>
- [ ] **(Owner: @name, due: YYYY-MM-DD)** Add runbook for <scenario>
- [ ] **(Owner: @name, due: YYYY-MM-DD)** Add metric/alert for <early detection>
- [ ] **(Owner: @name, due: YYYY-MM-DD)** Eval case covering <regression>

## Lessons Learned

<Что изменим в процессах/культуре. Не action items (те конкретные), а
принципы. Например: «provider updates надо отслеживать автоматически,
не полагаясь на user reports» или «cost alarms были настроены на daily,
а нужно на hourly».>

## Appendix

- Ссылки на traces: <trace_ids>
- Логи инцидента: <link to dashboard>
- Alert definition: <link>
- Чат обсуждения: <link>
````

## Шаги заполнения

### Шаг 1: Сразу после инцидента (в течение часа)

Пока свежо в памяти:
1. Запустите `scripts/generate_postmortem.py` — создаст пустой template с метаданными.
2. Заполните **Summary** и **Timeline** — это самое хрупкое, что забывается.
3. Прикрепите ссылки на traces, alerts, логи.
4. Назначьте Lead — тот, кто будет водить postmortem-встречу.

### Шаг 2: Через 24 часа

1. Заполните **Root Cause** — возможно, потребует дополнительного расследования.
2. Опросите участников — что они видели, что думали, что пробовали.
3. Сформулируйте **What Went Well / Poorly** — без имен, только процессы.

### Шаг 3: Встреча postmortem (в течение 48 часов)

1. Lead презентует документ.
2. Группа добавляет пропущенные детали.
3. **Action Items** назначаются с владельцами и сроками.
4. Финализируется **Lessons Learned**.

### Шаг 4: После встречи

1. Статус → **Final**.
2. Публикуется во внутреннем knowledge base.
3. Action items заносятся в трекер с дедлайнами.
4. Через 30 дней — follow-up: закрыты ли action items.

## Примеры постмортемов (типовые сценарии)

### Сценарий A: Model provider outage

```
Summary: 2026-07-20 с 10:23 до 11:15 UTC YandexGPT API отвечал 5xx на
~40% запросов. Failover на GigaChat не сработал. ~12K users получили
ошибку "Сервис недоступен". Cost impact: $0 (никаких лишних трат).
SLI: success rate упал до 60% на 52 минуты (SLO target 99%).

Root Cause:
1. YandexGPT deploy на их стороне в 10:20 UTC вызвал регрессию
2. Наш healthcheck проверял только HTTP 200, не semantic validity
3. Failover не сработал, т.к. healthcheck продолжал быть «зелёным»
4. Alarm на error rate сработал через 8 минут — это много
```

### Сценарий B: Prompt injection detected

```
Summary: 2026-07-19 обнаружена распределённая атака prompt injection
через user-generated контент. ~150 запросов за час от ~20 user_id
содержали паттерны «ignore previous instructions». Guardrail отбил все,
никаких leakage. Но alarms сработали только через 40 минут после
начала атаки.

Root Cause:
1. Guardrail rules корректно ловили известные паттерны
2. Alarm на guardrail rate baseline был настроен на 3× — это слишком высоко,
   атака началась медленно
3. Не было alert'а на «один и тот же user_id трижды за час»
4. Список атакующих user_id не передан в auth-слой автоматически
```

### Сценарий C: Cost spike

```
Summary: 2026-07-18 с 14:00 до 18:00 UTC средняя cost per session выросла
с $0.04 до $0.31 (8×). Никаких user-facing проблем. Daily budget
превышен на $340.

Root Cause:
1. Deploy в 13:45 обновил prompt template — добавил «подумай шаг за шагом»
2. Модель начала reasoning через 8-12 итераций вместо 2-3
3. Cost alarm был настроен на daily aggregate — сработал только в 18:00
4. В CI не было cost-регрессионного теста на новом prompt template
```

## Скрипт generate_postmortem.py

```bash
# Базовая генерация — сегодня, SEV-2
python3 skills/incident-postmortem/scripts/generate_postmortem.py \
    --title "YandexGPT outage, failover не сработал" \
    --severity SEV-2 \
    --output postmortems/2026-07-20-yandex-outage.md

# С заполнением lead
python3 skills/incident-postmortem/scripts/generate_postmortem.py \
    --title "Prompt injection attack" \
    --severity SEV-1 \
    --lead "@oncall-eng" \
    --duration "1h 12m"
```

Скрипт:
- Генерирует `incident_id` (YYYY-MM-DD-NNN).
- Подставляет текущую дату/время.
- Создаёт пустой template с правильной структурой.
- Выводит в stdout или в файл.

## Антипаттерны

- **«Случайно сломалось»** — без RCA. Всегда есть причина, даже если «никто не менял код» (model drift, infra change).
- **Action items без владельцев и сроков** — wishlist, а не действия.
- **Action items «улучшить коммуникацию»** — слишком размыто, никогда не будет сделано. Конкретизируйте.
- **Список виновных** — нарушает blameless принцип, люди перестают сообщать об инцидентах.
- **Закрытие постмортема до завершения action items** — постмортем без action items закрыт не должен быть.
- **Игнорировать повторяющиеся инциденты** — если 3-й раз одинаковая RCA, проблему в процессах, а не в каждом отдельном случае.
