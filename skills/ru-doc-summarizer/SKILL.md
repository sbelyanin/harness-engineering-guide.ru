# RU Doc Summarizer — Skill для саммаризации русскоязычных документов

Превращает длинный русскоязычный документ в структурированный report: решения, action items, ссылки, риски. Спроектировано под реалии русскоязычной технической и нормативной прозы — длинные периоды, канцелярит, неявные ссылки на смежные документы.

## Когда использовать

- Разобрать многостраничный регламент/ГОСТ/СТО под конкретную задачу
- Подготовить выжимку RFC или design-doc’а на русском для команды
- Превратить протокол совещания или тред в issue-трекере в список решений и action items
- Свести пул русскоязычных отзывов/тикетов в themes + частоты

## Специфика русскоязычных документов

Перед запуском учтите четыре особенности, которые ломают «английский» подход к саммаризации:

1. **Токен-ёмкость.** Русский текст расходует 1.2–1.5× токенов относительно английского при том же смысле. Бюджет chunk’ов ужимайте на 20–30%.
2. **Длинные периоды.** Одно предложение может содержать 4–6 подчинённых clauses; модель склонна выкидывать хвост. Просите явное декомпозирование.
3. **Канцелярит и пассив.** «Подлежит согласованию» = action item без владельца; «рекомендуется» ≠ «обязательно». Классифицируйте модальность явно.
4. **Неявные ссылки.** «Согласно п. 5.3 настоящего документа», «в соответствии с распоряжением №… от…» — без разрешения ссылок report бесполезен.

## Процесс (5 шагов)

### Шаг 1: Карта документа

**Цель:** понять структуру до саммаризации.

```
Вход: полный текст документа (или путь к файлу)
Действие:
  1. Извлечь иерархию заголовков (H1..H4, нумерованные разделы)
  2. Подсчитать объём по разделам (в токенах или знаках)
  3. Выявить разделы «по теме запроса» и «косвенные»
Выход: outline с оценкой релевантности каждого раздела
```

Если документ > контекстного окна — режьте на chunk’и по границам разделов, не по строкам.

### Шаг 2: Извлечение решений и обязательств

**Цель:** вытащить всё, что фиксирует обязательства.

Для каждого раздела соберите:

- **Решения** — что зафиксировано как принятое
- **Action items** — кто что должен сделать и к какому сроку
- **Ограничения** — что нельзя делать
- **Открытые вопросы** — что осталось нерешённым

**Модальность — обязательно:**

| Метка | Значение | Маркер в тексте |
|-------|----------|-----------------|
| `MUST` | обязательно | «обязан», «подлежит», «не допускается», «запрещено» |
| `SHOULD` | настоятельно рекомендуется | «рекомендуется», «следует», «целесообразно» |
| `MAY` | допустимо | «может», «допускается», «по усмотрению» |

### Шаг 3: Разрешение ссылок

**Цель:** превратить неявные ссылки в actionable-список.

```python
# Эвристики для извлечения русскоязычных ссылок
patterns = [
    r"согласно\s+п\.?\s*\d[\.\d]*",                    # согласно п. 5.3
    r"в\s+соответствии\s+с\s+(?!настоящ)",             # в соответствии с ...
    r"распоряжени\w+\s*№?\s*[\w\-/]+",                 # распоряжение № 123-р
    r"приказ\w*\s+(от|№)",                              # приказ от ... / № ...
    r"ГОСТ\s*Р?\s*[\d\-\.]+",                          # ГОСТ Р 12345-2020
    r"\bRFC\s*\d+",                                    # RFC 1234
]
```

Для каждой найденной ссылки определите:

- **Тип:** внутренняя (на этот же документ) / внешняя (на другой документ) / нормативная
- **Доступность:** есть ли в рабочей директории или нужно запросить
- **Критичность:** без неё решение не понять?

### Шаг 4: Риски и противоречия

**Цель:** подсветить, где саммари может быть ошибочным или документ внутренне противоречив.

Ищите:

- **Противоречия** — два раздела требуют взаимоисключающих действий
- **Устаревшие ссылки** — referenced-документ отменён/заменён
- **Нечёткие обязательства** — action item без владельца или дедлайна
- **Перекрытие с другими документами** — правило уже зафиксировано где-то ещё

### Шаг 5: Контрольный пересказ

**Цель:** убедиться, что саммари не выдумано.

- Для каждого пункта report’а укажите **source span** (раздел + диапазон строк)
- Поставьте **confidence**: 🟢 прямая формулировка / 🟡 парафраз / 🔴 вывод по умолчанию
- Любой пункт с confidence 🔴 должен сопровождаться цитатой-основанием

## Композитная оценка качества документа

Помимо саммари, оцените документ по 4 осям (0–2 балла, итого 8):

| Ось | 0 | 1 | 2 |
|-----|---|---|---|
| Полнота обязательств | Все action items без владельцев | Часть имеет владельцев | Все владельцы и сроки указаны |
| Согласованность | Внутренние противоречия | Локальные шероховатости | Полная согласованность |
| Ссылочная гигиена | Внешние ссылки не разрешаются | Часть ссылок разрешена | Все ссылки activatable |
| Исполнимость | Требования невыполнимы без доп. контекста | Требуют уточнений | Готовы к исполнению |

**Интерпретация:**
- 0–2: документ требует существенной доработки
- 3–5: пригоден, но с оговорками
- 6–8: зрелый, готов к исполнению

## Формат вывода

```
# Саммари: <заголовок документа>

## Метаданные
- Источник: <путь или URL>
- Объём: <N страниц / M токенов>
- Дата документа: <если есть>
- Оценка качества: X / 8

## Краткое резюме (3–5 предложений)
<суть документа для быстрого чтения>

## Решения
- [MUST] <решение 1> — (раздел 2.1, строки 45–52) 🟢
- [SHOULD] <решение 2> — (раздел 3.4, строка 110) 🟡

## Action items
| Владелец | Действие | Срок | Модальность | Источник |
|----------|----------|------|-------------|----------|
| ??? | согласовать план миграции | не указан | MUST | п. 5.3 🔴 |

## Ссылки, требующие разрешения
| Что | Тип | Где | Доступность |
|-----|-----|-----|-------------|
| распоряжение № 14-р от 2025-03-01 | внешнее | п. 2.4 | запросить |

## Риски и противоречия
- 🔴 Action item без владельца: «подлежит согласованию» (п. 5.3)
- 🟡 Section 4.2 и Section 7.1 расходятся в толковании срока хранения

## Что дальше
1. Запросить <внешние документы>
2. Уточнить владельцев у action items 🔴
3. Согласовать противоречия с автором документа
```

## Адаптация

- **Очень длинный документ (>128K токенов):** используйте map-reduce — саммаризуйте каждый раздел отдельно, затем сводите. См. `scripts/analyze.py` для чанкования.
- **Документ в PDF/DOCX:** извлеките текст с сохранением заголовков (pdfplumber, python-docx). Грубый `pdftotext` теряет иерархию — outline будет хромать.
- **Сканные ГОСТы (OCR):** прогоните OCRed-текст через корректор перед саммаризацией, иначе confidence будет мусорным.
