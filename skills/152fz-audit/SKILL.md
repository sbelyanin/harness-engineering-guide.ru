# 152ФЗ-Audit — Skill для аудита ПДн в логах AI-агента

Превращает сырой chat-лог (или любой текстовый артефакт agent-системы) в scored report: какие ПДн туда попали, насколько это критично по 152-ФЗ и что делать — маскировать, удалить, оставить.

## Когда использовать

- **Перед выкаткой eval-датасета** — убедиться, что в нём не осталось ПДн реальных пользователей
- **Перед передачей логов третьей стороне** (баг-репорт, внешняя команда, cloud-провайдер для fine-tuning)
- **При инциденте** — быстро оценить, что утекло и какой категории ПДн
- **Периодический compliance-ревью** memory-логов и session-историй
- **Перед публикацией** case-study / blog-поста с реальной перепиской

## Что считается ПДн

152-ФЗ определяет ПДн как **любую информацию, относящуюся к определённому физлицу**. Skill распознаёт:

| Категория | Уровень защищённости | Что ищет |
|-----------|----------------------|----------|
| Контактные | УЗ-1, УЗ-2 | email, телефон (РФ и intl), telegram-handle, привязанный к лицу |
| Идентификаторы | УЗ-1, УЗ-2 | ИНН, СНИЛС, ОМС, серия+номер паспорта, номер ВУ |
| Финансовые | УЗ-2 | номер карты (PAN), CVV, СБП-идентификаторы, ИИС-номера |
| Локационные | УЗ-1 | почтовый адрес с привязкой к лицу, координаты |
| Биометрические | УЗ-4 | упоминания голоса, отпечатков, фото-верификации |
| Специальные | УЗ-3 | здоровье, национальность, религия, судимость |
| Контекстные | зависит | «Иван Иванов, наш клиент из Москвы» — косвенные ПДн |

## Процесс аудита (6 шагов)

### Шаг 1: Загрузка и нормализация

**Цель:** привести лог к единому виду, отделив ПДн-содержимое от метаданных.

```
Вход: один из
  - .jsonl (по одной session на строку)
  - .json (массив сообщений с role/content)
  - .txt / .md (свободный текст)
  - директория с файлами выше
Действие:
  1. Распарсить в unified-формат: [{session_id, role, content, ts}, ...]
  2. Отделить метаданные (session_id, ts) — они не ПДн, но могут быть pseudonymized
  3. Сохранить исходные позиции (файл, строка) для source span’ов
Выход: normalized log + manifest
```

### Шаг 2: Детекция прямых ПДн

**Цель:** найти формальные идентификаторы regex’ами.

```python
# Паттерны прямого детектирования (упрощённо — см. scripts/analyze.py)
PATTERNS = {
    "email": r"\b[\w.+-]+@[\w-]+\.[\w.-]+\b",
    "phone_ru": r"(?:\+7|8)[\s\-]?\(?\d{3}\)?[\s\-]?\d{3}[\s\-]?\d{2}[\s\-]?\d{2}",
    "phone_intl": r"\+\d{1,3}[\s\-]?\(?\d{1,4}\)?[\s\-]?\d{3,}[\s\-]?\d{3,}",
    "inn": r"\b\d{10}|\d{12}\b",
    "snils": r"\b\d{3}-\d{3}-\d{3}\s?\d{2}\b",
    "passport": r"\b\d{4}\s?\d{6}\b",
    "card_pan": r"\b(?:\d[ -]*?){13,19}\b",
    "oms": r"\b\d{16}\b",
}
```

Для каждого совпадения: запишите `(session_id, field, value, position, pattern_name)`.

**Внимание:** `card_pan` даёт ложные срабатывания на длинных числах. Применяйте Luhn-проверку (см. скрипт).

### Шаг 3: Детекция косвенных ПДн

**Цоль:** найти упоминания лиц без явных идентификаторов.

Прямые regex’ы не ловят «Иван Иванов оформил подписку вчера». Для этого используется LLM-классификатор с жёстким prompt’ом:

```
Проанализируй следующий фрагмент chat-лога. Найди упоминания персональных данных
физических лиц. Классифицируй по категориям: ФИО, должность+компания (если человек
определяем), адрес,健康状况, семейное положение, религия, национальность.

Верни JSON:
{
  "findings": [
    {
      "category": "...",
      "snippet": "...",
      "identifiability": "direct | indirect",
      "confidence": 0.0-1.0
    }
  ]
}

Фрагмент:
<TEXT>
```

Прогоняйте по chunk’ам, не по всему логу — иначе модель теряется и галлюцинирует.

### Шаг 4: Классификация риска

**Цель:** оценить критичность каждой находки по 152-ФЗ.

```python
def classify_risk(finding: dict) -> RiskLevel:
    cat = finding["category"]
    ident = finding.get("identifiability", "indirect")

    # Биометрия и специальные — автоматически УЗ-4 / УЗ-3
    if cat in {"biometric", "health", "religion", "nationality"}:
        return RiskLevel.UZ4 if cat == "biometric" else RiskLevel.UZ3

    # Финансы — УЗ-2
    if cat in {"card_pan", "cvv", "sbp_id"}:
        return RiskLevel.UZ2

    # Контактные + идентификаторы — зависит от связки
    if cat in {"email", "phone", "inn", "snils", "passport"}:
        return RiskLevel.UZ2 if ident == "direct" else RiskLevel.UZ1

    return RiskLevel.UZ1
```

### Шаг 5: Агрегация и оценка поверхности

**Цель:** понять, насколько широко ПДн разошлись по логу.

Метрики:

- **Total findings** — сколько всего ПДн-фрагментов найдено
- **Unique subjects** — сколько уникальных лиц затронуто (оценивается по хешам identifier-полей)
- **Sessions affected** — в скольких session встречается хотя бы одно ПДн
- **Categories distribution** — разбивка по категориям
- **Highest risk** — самый высокий УЗ среди находок
- **Density** — ПДн / 1000 токенов текста (для бенчмарка)

### Шаг 6: Рекомендации по remediation

Для каждой находки предложите действие:

| Ситуация | Действие |
|----------|----------|
| Card PAN с прошёдшим Luhn | 🔴 удалить немедленно, уведомить security |
| Биометрия в логе | 🔴 удалить, оценить необходимость лицензии |
| ИНН / СНИЛС в production-логе | 🟠 pseudonymize, проверить согласие субъекта |
| Email / телефон в логе | 🟡 pseudonymize, ввести masking в logging-pipeline |
| ФИО в контексте тикетты | 🟢 допустимо, если есть согласие; иначе pseudonymize |
| Косвенные упоминания, confidence <0.6 | ⚪ игнорировать (вероятно false positive) |

## Композитная оценка лога

Каждый лог оценивается 0–16 баллами (4 оси × 0–4):

| Ось | 0 | 4 |
|-----|---|---|
| Объём ПДн | нет | >100 находок |
| Уровень защищённости | нет УЗ+ | УЗ-4 |
| Распространение | 1 session | >50 sessions |
| Зрелость pipeline masking | нет masking | full pseudonymization |

**Интерпретация:**
- 0–4: ✅ лог пригоден к использованию (с оговорками)
- 5–9: ⚠️ требует pseudonymization перед передачей третьим
- 10–16: 🚨 критический — удаление + расследование инцидента

## Адаптация

- **Очень большой лог (>1M сообщений):** используйте map-reduce — каждый session отдельно, затем агрегация. См. `scripts/analyze.py` для чанкования.
- **Зашифрованные логи:** декодируйте перед аудитом; сам audit-log не должен содержать ПДн — только хеши.
- **Кастомные идентификаторы** (например, внутренний account_id формата `ACC-XXXX`): добавьте паттерны в `PATTERNS` и укажите уровень защищённости через `classify_risk`.

## Антипаттерны

- **Удалять логи без аудита** — нарушает требования к журналу доступа (ст. 19 п. 4); сначала audit, потом masking
- **Полагаться только на regex** — косвенные ПДн (ФИО + должность) ловятся только LLM-классификатором
- **Маскировать всё подряд** — лог становится нечитаемым, снижается debug-value; классифицируйте риск и маскируйте по необходимости
- **Не псевдонимизировать eval-датасеты** — ПДн «обезличенного» пользователя часто восстанавливаются через корреляцию
