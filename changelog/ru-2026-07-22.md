### A7. Пунктуация и типографика

- **ASCII `"..."` → `«...»` (ёлочки)** в прозе. 23 замены. Code-блоки и YAML frontmatter не тронуты. Использован CommonMark-совместимый state-machine парсер, корректно обрабатывающий nested code-fences.
- **Неразрывные пробелы (NBSP, U+00A0)** расставляются автоматически при рендере HTML через новую функцию `typographRussian()` в `site/lib/content.ts`:
  - после одиночных предлогов/союзов (и, а, но, в, с, к, у, о, об, на, по, до, из, от, за, для, или, что, как, не, ни, это, же, ли, бы, я, он, мы, вы, ...) — 31 предлог;
  - между числом и короткой кириллической единицей/словом (10 кг, 5 минут, 128K токенов).
  - `<code>` и `<pre>` блоки пропускаются — NBSP не попадает в копируемый код.
  - Markdown-исходники остаются с обычными пробелами (правила применяются только в HTML-выводе).

## Инженерные нюансы

### State-machine для code-fences вместо regex

Первая попытка использовать regex `r"(```\w*\n[\s\S]*?\n```)"` для разделения markdown на prose/code сегменты **ломала nested code-fences** — JSON-примеры внутри внешнего code-блока (` ``` ` → текст → ` ```json ` → JSON → ` ``` ` → ` ``` `) обрабатывались как prose, и ASCII-кавычки внутри JSON заменялись на «ёлочки», ломая валидность JSON.

Решение: простой state-machine построчный парсер, который открывает code-fence при `^\`\`\`` и закрывает только при `^\`\`\`+$` (ровно backticks, без info-string). Это совместимо с CommonMark spec.

### Защита `<code>`/`<pre>` при NBSP-обработке

NBSP-функция в `content.ts` использует `split(/(<code...>|<pre...>)/)` для выделения inline-code и pre-блоков, пропуская их. Проверено: в `your-first-harness` (20 NBSP) ни один не попал внутрь `<code>` или `<pre>`.
