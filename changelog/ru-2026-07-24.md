# 2026-07-24 — Завершение Этапа 9

Полный цикл observability для harness: от инструментации модели до runbook'а на дежурстве. LLM-системы принципиально недетерминированы — классический RED мониторинг (Rate/Errors/Duration) здесь ломается, потому что HTTP 200 с галлюцинацией или refusal'ом технически «успех», но user-impact негативный.

Этап 9 дал читателю **пять статей**, **два skills** и **страницу-пример** на сайте. Всё, что нужно для запуска harness в production.

[Observability](/guide/observability)** — введение: почему классического мониторинга недостаточно для LLM, четыре сигнала (metrics + logs + traces + **evals** — последний уникален для harness). - **I5. [Incident Runbook](/guide/incident-runbook)** — типовые инциденты (model outage, tool breakage, prompt injection, cost spike, context-window exhaustion, eval regression) с diagnosis/remediation для каждого, postmortem template, blameless-принципы.

## Инженерные нюансы

### REDCoT — расширение RED для LLM

Классическая RED (Rate, Errors, Duration) создавалась для детерминированных HTTP-сервисов. Harness ломает её: один запрос стоит 0.001$, другой — 0.50$; один идёт 800ms, другой — 25s (chain-of-thought через tool calls). **REDCoT = RED + Cost + Tokens** — единственная рабочая модель для LLM-monitoring. Cost и Tokens — метрики первого класса, не «финансовый отчёт раз в месяц».

### Tail sampling для traces

Сохранять 100% трейсов в production обычно не нужно (storage-cost). Стратегия: **head sampling 5–10% для здоровых запросов + 100% для ошибочных и медленных**. Tail sampling (через OTel Collector) даёт гибкость: решение принимается по итогам трейса, а не по его началу. Для harness'а это критично — agent-loop может выглядеть здоровым в начале, но зависнуть на 7-й итерации.

### Blameless postmortem — не софт-скилл, а системная инженерия

В постмортеме **имена людей как «причина» — это red flag**. Если в действии «dev X залил баг» нет системной причины (CI не покрыл, review не нашёл, deploy без feature flag) — вы не исправили процесс, вы назначили козла отпущения. В следующий раз X не сообщит об инциденте. Blameless-культура — не про «доброту», а про **не скрывать инциденты**, чтобы учиться на них.

### GenAI semantic conventions — не изобретайте свои

OTel имеет [готовую спецификацию](https://opentelemetry.io/docs/specs/semconv/gen-ai/) для GenAI-вызовов: `gen_ai.system`, `gen_ai.request.model`, `gen_ai.usage.input_tokens`, и т.д. Используйте стандартные имена — это даёт совместимость с любым observability-vendor'ом и шаблонные dashboard'ы в Grafana. Свой формат атрибутов — это технический долг с первого дня.
