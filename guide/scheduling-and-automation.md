# Scheduling и автоматизация

> **Главный инсайт:** Agent, который отвечает только когда к нему обращаются, — это чат-бот; Agent, который действует по расписанию, по событию и по собственной инициативе, — это productivity-система.

## Зачем нужен Scheduling

Большинство людей знакомится с агентами как с разговорным интерфейсом — вы печатаете, он отвечает. Эта ментальная модель ограничивает ценность «поисковиком побыстрее». Настоящий unlock случается, когда агенты делают работу *без запроса*.

Сравните:

- **Реактивно:** «Саммаризуй мои непрочитанные письма». → Агент отвечает.
- **По расписанию:** Каждое утро в 8:00 агент читает инбокс, фильтрует по срочности, форматирует дайджест и пушит в ваш Slack-канал. Вы просыпаетесь с брифингом, которого не заказывали.

Второй паттерн множится. Одна запланированная задача экономит пять минут в день. Десять — меняют характер работы. Scheduling превращает агента из инструмента, который вы используете, в систему, работающую на вас.

Эта глава разбирает примитивы, которые делают запланированную работу агента возможной внутри Harness — runtime-среды, где хостятся Tools, Context, memory и execution-lifecycle агента.

## Виды запланированной работы агента

Не вся автоматизация идёт по одним часам. Четыре паттерна покрывают ландшафт:

### One-shot-таймеры

Единичное отложенное выполнение. Пользователь говорит «напомни через 20 минут» или «отправь follow-up в 15:00». Harness создаёт таймер, срабатывает один раз и выбрасывает.

Ключевые свойства:
- Срабатывает ровно один раз
- Часто создаётся разговорно («напомни…»)
- Короткоживущий — минуты-часы, редко дни
- Без логики повторений

One-shot-таймеры — простейший scheduling-примитив. Нужны только timestamp и payload.

### Периодические Cron'ы

Cron (по имени Unix-демона `cron`) — задача, запускающаяся по повторяющемуся расписанию, заданному cron-выражением — компактным синтаксисом для минуты, часа, дня, месяца и дня недели.

Примеры:
- `0 8 * * *` → каждый день в 08:00
- `0 9 * * 1` → каждый понедельник в 09:00
- `*/30 * * * *` → каждые 30 минут

Cron'ы — backbone agent-автоматизации. Дневные дайджесты, еженедельные отчёты, периодический мониторинг — всё это территория Cron.

### Event-триггеры

Не все расписания основаны на времени. Часть работы должна срабатывать в ответ на внешние события:

- Открыт новый Pull Request → породить review-агента
- Развёртывание завершено → прогнать smoke-тесты и отчитаться
- Пришло письмо от VIP-отправителя → саммаризовать и алертнуть

Event-автоматизация требует источника событий (webhook, polling или платформенная интеграция) и механизма диспетчеризации, создающего ход агента при срабатывании события. Здесь Harness работает не по часам — по сигналам.

### На базе Heartbeat

Heartbeat — периодический poll, где Harness инъецирует промпт в *главную session* агента через регулярный интервал (обычно каждые 15–60 минут). Агент читает промпт, решает, что проверить, и либо действует, либо молчит.

В отличие от Cron, создающего изолированные выполнения, Heartbeat крутится внутри текущего диалога. У агента есть доступ к недавним сообщениям, user-context и состоянию session. Это делает Heartbeats идеальными для батчинга нескольких лёгких проверок в один ход.

Типичный Heartbeat-промпт:

```
Read HEARTBEAT.md if it exists. Follow it strictly.
If nothing needs attention, reply HEARTBEAT_OK.
```

Агент может проверить почту, глянуть календарь, ревьюить уведомления — всё за один ход, со всем разговорным context.

## Паттерны реализации Cron

Cron — самый структурированный scheduling-примитив. Чтобы сделать его правильно, нужно внимание к четырём измерениям: определение расписания, targeting session, тип payload и доставка.

### Определение расписания

**Cron-выражения** используют стандартный пятиполевой формат:

```
┌───────── минута (0–59)
│ ┌─────── час (0–23)
│ │ ┌───── день месяца (1–31)
│ │ │ ┌─── месяц (1–12)
│ │ │ │ ┌─ день недели (0–7, 0 и 7 = воскресенье)
│ │ │ │ │
* * * * *
```

**Работа с таймзонами** — самый частый источник багов в Cron. Правило простое:

> **Хранить UTC. Показывать локальное.**

Внутри Harness оценивает все cron-выражения по UTC. Показывая пользователю расписание, конвертируйте в его локальный часовой пояс. Когда пользователь говорит «каждый день в 8 утра», Harness должен знать, *какое именно* 8 утра — спросите таймзону, если неизвестно, храните UTC-эквивалент и подтверждайте обратно в локальном времени.

```
User: "Run my digest every morning at 8am"
Agent: "Got it — daily digest at 8:00 AM Asia/Shanghai (0:00 UTC). ✓"
```

Это устраняет неоднозначность и переживает переходы на летнее время (для таймзон, где они есть).

### Targeting Session

Когда Cron срабатывает, *где* работает агент? Две модели:

**Изолированная session** — Cron порождает независимый ход агента со свежим context window. Без истории диалога. Без предыдущих сообщений. Агент стартует чистым, выполняет задачу и терминируется.

Плюсы:
- Нет загрязнения истории — основная session остаётся чистой
- Предсказуемый context — агент видит только то, что даёт Cron
- Параллелизируется — несколько cron-задач могут идти concurrently без интерференции
- Конфигурируемо — каждая задача может использовать другую модель или уровень thinking

Используйте изолированные session для самостоятельных задач: дайджесты, отчёты, monitoring-проверки, сбор данных.

**Главная session** — Cron инъецирует контент в текущую активную session пользователя. Агент видит полную историю диалога и может ссылаться на недавние сообщения.

Плюсы:
- Разговорный context — агент знает, что вы обсуждали
- Непрерывность — может продолжать ранние треды
- Меньше оверхеда — не надо создавать новую session

Используйте инъекцию в главную session умеренно. Каждая инъекция добавляет в context window, и лишнее раздует её (см. Антипаттерны ниже).

### Типы Payload

Что именно доставляет Cron при срабатывании?

**agentTurn** — запускает полный цикл выполнения агента. Агент получает промпт, имеет доступ ко всем своим Tools и Skills, принимает решения, вызывает API, читает файлы и выдаёт результат. Самый мощный тип payload.

Пример: дневной digest-Cron срабатывает с payload `agentTurn`. Агент читает письма через IMAP, проверяет календарь, тянет погоду, форматирует всё в брифинг и пушит в канал. Полная автономия.

**systemEvent** — инъецирует текстовое сообщение в session без запуска полного хода агента. Как записка, брошенная в диалог. Агент увидит её при следующем ходе (или Heartbeat), но не действует немедленно.

Пример: webhook получает уведомление о развёртывании и инъецирует «Deploy v2.3.1 completed successfully» как systemEvent. Агенту не нужно действовать — но информация доступна, если пользователь спросит.

### Доставка

Куда идут результаты?

- **Анонс в канал** — вывод агента пушится в конкретный канал сообщений (Slack, Discord, Feishu-группа и т.д.). Самая частая доставка для дайджестов и алертов.
- **Webhook** — результаты POST'ятся на HTTP-эндпоинт для интеграции с внешними системами.
- **Никакая (silent)** — Cron отрабатывает, но не производит внешнего вывода. Полезно для фоновых maintenance-задач вроде очистки memory или предзагрузки данных.

Конфигурацию доставки нужно задавать при создании Cron и пинить к исходной session. Если пользователь создаёт Cron в Feishu-группе, результаты должны идти в эту группу — а не следовать за пользователем туда, где он недавно общался.

## Реальные примеры

### LangSmith Deployments Cron

Платформа LangSmith от LangChain добавила cron-задачи для feature Deployments (см. [документацию LangSmith Cron Jobs](https://docs.langchain.com/langsmith/cron-jobs)). Модель:

1. Создаёте Deployment (хостимый LangGraph-агент)
2. Определяете cron-расписание
3. На каждый тик система создаёт новый thread и шлёт фиксированный input

Подход чистый и предсказуемый. Каждое выполнение получает один input, свежий thread и производит трассируемый вывод в мониторинг-UI LangSmith.

**Сильные стороны:**
- Простая ментальная модель — один input, свежий thread, каждый раз
- Полная наблюдаемость через LangSmith-traces
- Интеграция с state-управлением LangGraph

**Ограничения:**
- Фиксированный input на Cron — нельзя динамически менять, что делает агент, по context
- Thread-per-execution — нет непрерывности между запусками
- Привязка к инфраструктуре LangSmith Deployments

### Harness-native Cron

Паттерн Harness даёт более богатую модель:

1. Определяете Cron с расписанием, targeting session, payload и доставкой
2. На каждый тик Harness порождает изолированную session (или инъецирует в главную)
3. У агента полный доступ к Tools — может читать файлы, вызывать API, искать в вебе, использовать любой установленный Skill
4. Результаты доставляются в настроенный канал

```yaml
# Концептуальное определение Cron
schedule: "0 8 * * *"          # 8:00 UTC
timezone: "Asia/Shanghai"       # Референс отображения
session: isolated               # Свежий context каждый запуск
payload: agentTurn              # Полное выполнение агента
model: claude-sonnet            # Может отличаться от дефолтной
delivery:
  type: announce
  channel: feishu-group-abc     # Запинено на исходную session
prompt: |
  Generate today's morning briefing:
  1. Check unread emails for urgent items
  2. List today's calendar events
  3. Summarize overnight GitHub notifications
  Format as a concise digest.
```

**Ключевые отличия от LangSmith:**
- Динамические промпты — агент решает, что делать, а не просто обрабатывает фиксированный input
- Полный доступ к Skill — может использовать любой Tool, который даёт Harness
- Выбор модели на задачу — дешевле для простых проверок, сильнее для анализа
- Гибкая доставка — канал, webhook или silent

### Паттерн дневного дайджеста

Самый частый use-case Cron. Поток:

```
Cron срабатывает (8:00 локально)
  → Агент порождается в изолированной session
  → Читает почту (IMAP Tool)
  → Читает календарь (Calendar Tool)
  → Проверяет погоду (Weather Skill)
  → Проверяет GitHub-уведомления (GitHub Skill)
  → Форматирует дайджест с приоритетами и хайлайтами
  → Пушит в предпочтительный канал пользователя
  → Session терминируется
```

Множитель ценности: это гоняется каждый день, вспомнит пользователь попросить или нет. Информация приходит предобработанной и отформатированной.

### Паттерн мониторинга

Cron, который проверяет метрики и алертит только при пробитии порогов:

```
Cron срабатывает (каждые 15 минут)
  → Агент проверяет времена ответа API
  → Сравнивает с порогом (p99 < 500ms)
  → Если здорово: silent (без доставки)
  → Если деградировало: алерт в ops-канал с деталями
  → Session терминируется
```

Ключевое проектное решение: **алерт по исключению, не по каждой проверке.** Monitoring-Cron, пушащий «всё ок» каждые 15 минут, — это шум. Молчащий до сбоя — это сигнал.

## Heartbeat vs Cron: когда что использовать

И Heartbeat, и Cron дают периодическую работу агента. Они служат разным целям.

| Измерение | Heartbeat | Cron |
|-----------|-----------|------|
| **Тайминг** | Приблизительный (drift допустим) | Точный (по расписанию) |
| **Context** | Главная session (полная история) | Изолированный (свежий context) |
| **Проверок за запуск** | Несколько (батч) | Фокус на одной задаче |
| **Влияние на session** | Добавляет в main context window | Без влияния на главную session |
| **Конфигурация** | Редактировать HEARTBEAT.md | Явное определение Cron |
| **Модель** | Использует модель session | Можно указать на задачу |

**Используйте Heartbeat, когда:**
- Хотите батчить 3–5 лёгких проверок (почта + календарь + уведомления) в один ход
- Агенту нужен разговорный context, чтобы решить, что важно
- Точность тайминга не важна (каждые ~30 минут — ок)
- Хотите сократить суммарные API-вызовы, объединяя периодические проверки

**Используйте Cron, когда:**
- Важен точный тайминг («ровно 9:00 каждый понедельник»)
- Задача должна работать изолированно, не загрязняя основную session
- Нужна другая модель или thinking-бюджет под задачу
- Вывод должен идти в конкретный канал без затрагивания основной session
- One-shot-напоминания («напомни через 20 минут»)

**Правило:** если у вас пять периодических проверок и ни одна не требует точного тайминга, положите их в HEARTBEAT.md вместо пяти cron-задач. Если одна задача должна гоняться ровно в 9 утра с конкретной моделью и доставкой в конкретный канал — это Cron.

## Антипаттерны

### Polling-циклы

```python
# ❌ Не делайте так
while True:
    result = check_something()
    if result.changed:
        notify()
    sleep(60)
```

Это сжигает постоянный процесс, тратит compute и обходит scheduling-инфраструктуру Harness. Используйте Cron с минутным интервалом. Harness управляет lifecycle, retry и очисткой ресурсов.

### Загрязнение главной Session

Слишком много systemEvent-инъекций в главную session раздувает context window. Каждая инъекция добавляет токены. За день частых инъекций context агента занят системным шумом, а не полезным диалогом.

**Симптом:** ответы агента замедляются, становятся менее связными или теряют нить разговора.

**Фикс:** используйте изолированные session для высокочастотной работы. Оставьте инъекцию в главную session для редких высокосигнальных событий, которые пользователь должен видеть в context.

### Без таймаута

Cron-задача, способная работать бесконечно, — утечка ресурсов. Если агент завис (ждёт API, застрял в reasoning-цикле), задача висит вечно.

**Фикс:** у каждой cron-задачи должен быть таймаут. Разумный дефолт — 2–5 минут для простых задач, 10–15 для сложных. Если задача не укладывается в окно, она должна упасть громко — а не висеть молча.

### Дублирование доставки

```yaml
# ❌ Cron анонсирует в канал И агент вручную шлёт сообщение
delivery:
  type: announce
  channel: slack-general
prompt: |
  Check metrics and send results to #general  # Агент тоже шлёт!
```

Если Cron настроен анонсировать результаты в канал, а промпт агента тоже велит ему слать сообщение в тот же канал, пользователь получает тот же контент дважды.

**Фикс:** выберите один механизм доставки. Либо Harness анонсирует, либо агент шлёт вручную — никогда оба.

## Чек-лист проектирования

Перед созданием любой запланированной работы агента:

1. **Таймзона** — знаете ли таймзону пользователя? Если нет — спросите.
2. **Тип session** — нужен ли разговорный context (main) или чистое выполнение (isolated)?
3. **Доставка** — куда идут результаты? Запиньте на исходную session.
4. **Таймаут** — какое максимально приемлемое время выполнения?
5. **Failure-мод** — что происходит при провале задачи? Молча? Алерт? Retry?
6. **Частота** — это лучше как Cron или как запись в HEARTBEAT.md?
7. **Дедупликация** — точно результаты не доставятся дважды?

Если правильно закрыть эти семь пунктов, запланированная работа агента будет надёжной, предсказуемой и действительно полезной.

## Резюме

Scheduling — то, что отличает разговорную новинку от productivity-системы. Harness даёт четыре примитива — one-shot-таймеры, периодические Cron'ы, event-триггеры и Heartbeat'ы — каждый под свою форму автоматизации.

Архитектурные выборы важны: хранить UTC, показывать локальное. Изолировать session по умолчанию. Ставить таймауты. Пинить доставку на исходный канал. Батчить лёгкие проверки в Heartbeat вместо множества Cron'ов.

Агент, действующий по собственной инициативе, по расписанию, без запроса, — это уже не чат-бот. Это инфраструктура.
