# Agent Teams: параллельные Claude строят настоящее ПО

Большинство agent-демо выглядят впечатляюще и не делают ничего полезного. Todo-приложение. Скрейпер todo. «Персональный research-ассистент», саммаризующий три веб-страницы.

А потом кто-то втыкает 16 Claude в цикл, и они собирают C-компилятор, способный скомпилировать ядро Linux.

Эта статья — про второе и конкретно про то, как выглядел harness под ним. Пройдёмся по архитектуре из проекта Николаса Карлини *«Building a C compiler with a team of parallel Claudes»* ([Anthropic Engineering](https://www.anthropic.com/engineering/building-c-compiler), [GitHub](https://github.com/anthropics/claudes-c-compiler)), вытащим проектные принципы, которые реально важны, и переведём их в паттерны, переиспользуемые в вашем harness.

---

## Главный инсайт

> **16 экземпляров Claude, работая параллельно около двух недель, произвели C-компилятор на 100 000 строк на Rust. Он компилирует Linux 6.9 на x86, ARM и RISC-V. Проходит 99% GCC torture-test-сьюта. Компилирует QEMU, FFmpeg, SQLite, Postgres и Redis.**

Перечитайте. И ещё раз.

Это не игрушка. Это реальный production-компилятор, написанный командой LLM в условиях куда более жёстких, чем терпела бы человеческая команда: без доступа в интернет, без готовых compiler-крейтов, без копирования из LLVM или tcc — только стандартная библиотека Rust и то, что Claude могла вывести из C-spec и своего обучения. Clean-room-реализация, сделанная агентами.

Интересный вопрос не «может ли LLM написать компилятор». Мы уже знали, что может написать *части*. Интересный вопрос:

**Какой harness превращает 16 stateless, забывчивых LLM-вызовов в слаженную инженерную команду, выдающую 100 000 строк кода?**

## Архитектура Ralph-Loop

Сердце harness смущающе просто. Каждый агент — bash-цикл:

```bash
#!/usr/bin/env bash
# agent.sh — run inside a Docker container
while true; do
  claude --dangerously-skip-permissions -p AGENT_PROMPT.md
done
```

Вот и всё. Это весь агент.

Никакого оркестратора. Никакого демона очереди задач. Никакого планировщика. Никакой шины сообщений. Когда одна Claude-session заканчивается — успехом, провалом или коллапсом context window — цикл стартует свежую Claude с тем же файлом промпта. Промпт велит новой Claude посмотреть репо, понять, что делать дальше, и сделать.

Этот стиль иногда называют «Ralph loop» (в честь Ральфа Виггама «I'm helping!»): тупой внешний цикл пинает умный внутренний процесс, пока работа не будет done.

Единственное вмешательство супервизора:

```bash
# kill the fleet
pkill -9 bash
```

Это весь протокол остановки. Никакого graceful drain, никакого «пожалуйста, доделай текущую задачу». Вы SIGKILL'ите bash-циклы, и агенты умирают на полумысли. В следующий старт они подхватывают с того места, где оставил `git pull`.

### Почему это работает

Ralph loop — правильная форма для agent-работы по трём причинам:

1. **LLM всё равно stateless между session.** Притворяться, что они долгоиграющие процессы, только прячет коллапс context window за всё более отчаянными трюками.
2. **Рестарт бесплатен.** Свежая Claude, читающая репо, дешевле залежалой Claude, забившей свой context тремя часами провальных экспериментов.
3. **Изоляция падений идёт бесплатно.** Если session съезжает с рельсов, следующая не наследует обломки — она наследует git-состояние, единственный источник правды.

## Координация через Git (без оркестратора)

Вот часть, которая удивляет большинство людей: **нет центрального оркестратора.** Никто не раздаёт задачи. Никто не знает, кто чем занят. Нет дашборда с «agent-7 сейчас работает над struct-packing».

Вместо этого флот координируется через два примитива:

1. **Голый git-репо**, в который каждый контейнер push'ит и pull'ит.
2. Директория `current_tasks/` с **lock-файлами** — по одному на каждую task-in-progress, записывается как `current_tasks/<task-name>.txt`.

Промпт каждого агента говорит ему, по сути:

> «Посмотри на открытые TODO, падающие тесты и progress-файлы. Выбери самую очевидную следующую вещь, над которой никто сейчас не работает. Положи lock-файл. Сделай работу. Закоммить. Удали lock. Выйди.»

Lifecycle одной session:

```
 ┌──────────────────────────────────────────────────────────────┐
 │  1. git clone / pull                                         │
 │  2. сканировать репо → выбрать задачу T                      │
 │  3. git add current_tasks/T.txt   ← заявить lock             │
 │  4. git push                      ← опубликовать заявку      │
 │  5. работать над T (код, тесты, коммит)                      │
 │  6. git pull --rebase             ← забрать изменения пиров  │
 │  7. разрешить мержи / перегнать тесты                         │
 │  8. git push                      ← отгрузить                │
 │  9. git rm current_tasks/T.txt    ← снять lock               │
 │ 10. git push                                                 │
 │ 11. exit → Ralph loop порождает новую Claude в новом контейнере │
 └──────────────────────────────────────────────────────────────┘
```

### Гонка и почему побеждает git

Два агента просыпаются одновременно. Оба смотрят на открытые задачи. Оба решают, что самое очевидное — «починить segfault в codegen/x86.rs». Оба пишут `current_tasks/fix-segfault-x86.txt`. Оба `git push`.

Один push выигрывает. Другой отклонён — `non-fast-forward`. Тот агент делает `git pull --rebase`, видит, что lock уже существует, понимает, что опоздал, и идёт выбирать другую задачу.

Вот ключ: **модель конкурентности git — это же модель конкурентности agent-команды.** Локи, мержи, разрешение конфликтов — всё бесплатно, потому что git уже это решает. Не надо строить distributed-systems-поверх LLM; вы наследуете его из 2005-го.

## Принцип дизайна 1: пишите тесты для Claude, а не для себя

Это урок, который я выбил бы на лбу каждого, кто строит agent-harness.

Когда человек читает падающий тест, он получает многое бесплатно: может глянуть 500 строк вывода, заметить подозрительный паттерн, держать гипотезу в голове и пойти дебажить. LLM в цикле не умеет ничего из этого хорошо. Ваш test-harness нужно проектировать под её ограничения, а не под ваши.

### Загрязнение Context Window

Дефолтное поведение большинства test-раннеров — «вывалить всё в stdout». Для человека, скроллящего терминал, это ок; для Claude — катастрофа. Тысяча строк тест-вывода сжёвывает context window, и к моменту, когда Claude рассуждает о баге, она забыла промпт.

**Фикс:** сделайте test-harness крайне лаконичным по умолчанию, а многословность — в файлы.

```python
def run_test(name: str) -> TestResult:
    result = execute(name)
    # Вывод для человека (и LLM): одна строка.
    if result.ok:
        print(f"PASS {name}")
    else:
        print(f"ERROR {name}  see logs/{name}.log")
    # Всё остальное — на диск.
    write(f"logs/{name}.log", result.full_output)
    return result
```

Теперь Claude видит:

```
PASS  tests/struct_layout_01
PASS  tests/struct_layout_02
ERROR tests/struct_layout_03  see logs/struct_layout_03.log
PASS  tests/struct_layout_04
```

Claude может `grep ERROR` по прогону, выбрать один провал и `cat` только нужный лог. Context остаётся маленьким. Внимание сфокусированным.

### Маркер `ERROR`

Заметьте литерал `ERROR`. Это намеренно. Согласованный, greppable-маркер — одна из самых высоко-ROI-конвенций в любом agent-harness. Она значит, что Claude может навигировать тысячи тест-результатов одним `grep -l ERROR logs/`. Не пишите «failed» в одном месте, «FAIL» в другом и «❌» в третьем. Выберите одно. Ставьте везде.

### Слепота ко времени

У Claude нет wall-clock. Она не отличает «этот тест медленный» от «этот тест завис». Если разрешить агенту гонять полный тест-сьют каждую session, половина агентов просидит 40 минут, решит, что ничего не происходит, убьёт сьют и попробует дикий рефакторинг.

**Фикс:** добавьте subsample-режим `--fast`, гоняющий 1–10% тестов, детерминированно выбранных на агента:

```python
def select_tests(all_tests: list[str], agent_id: str, fraction=0.05):
    # Один агент → один subsample в рамках session.
    # Разные агенты → разные subsamples, так что флот покрывает
    # больше, чем любой одиночный агент.
    rng = seeded_random(agent_id)
    k = max(1, int(len(all_tests) * fraction))
    return rng.sample(all_tests, k)
```

Тут важны два свойства:

- **Детерминированность на агента**: один и тот же агент, дважды запуская одну команду, видит одни результаты. Иначе Claude не может рассуждать, починило ли её изменение что-то.
- **Различие по флоту**: каждый агент зондирует свой угол тест-сьюта, и объединённое покрытие куда выше, чем у любого одиночного агента.

### Progress-файлы и README

Единственный долго живущий кусок memory — файловая система. Агенты Карлини поддерживают:

- `README.md` в корне каждого модуля: *что это делает, что работает, что нет*.
- `PROGRESS.md` и per-area progress-файлы: *что пробовали, что упало, какая текущая теория*.

Это не для людей. Это для следующей Claude, которая проснётся. Когда вы трактоваете progress-файлы как **послания будущему себе**, они становятся самым ценным артефактом в репо.

## Принцип дизайна 2: декомпозиция задач под параллелизм

Параллелизм — не факт о команде. Это факт о *работе*.

### Лёгкий режим: много независимых провалов

Счастливый случай для agent-команды — тест-сьют с кучей независимых падающих тестов. 200 падающих тестов, 16 агентов, каждый берёт один, чинит, коммитит, идёт дальше. Минимум конфликтов. Линейное масштабирование. Все довольны.

```
┌───────────────────────┐
│  200 падающих тестов   │
├───────────────────────┤
│  agent-1 → тест 47    │
│  agent-2 → тест 113   │
│  agent-3 → тест 2     │
│  agent-4 → тест 88    │
│   ...                 │
└───────────────────────┘
```

### Сложный режим: одна монолитная сборка

Теперь представьте, что цель — *скомпилировать ядро Linux*. Одна задача. Одна сборка. Один падающий артефакт. 16 агентов все начинают читать исходники ядра и предлагать фиксы. Они перезаписывают друг друга. Спорят, чинить баг в парсере или в codegen. Репо становится демо-дерби.

Больше агентов теперь *хуже*. Закинуть больше Claude в одну неделимую проблему — способ сжечь $20K за полдня без результата.

Фикс — не «добавить оркестратора». Фикс — **декомпозируйте монолит в независимые провалы**. Что подводит к самой хитрой идее во всём проекте.

## GCC-as-Oracle: бисекция ядра

Когда команда попробовала скомпилировать ядро Linux своим новым компилятором, всё поломалось эффектно и запутанно. Сотни файлов вносили вклад в один сломанный образ. Никто — ни Claude, ни человек — не мог посмотреть на «ядро не загрузилось» и сказать, какой из 20 000 C-файлов вскрыл баг.

Ход: **использовать GCC как известный-хороший oracle и позволить Claude компилировать только подмножество**.

```
┌────────────────────────────────────────────────────────────┐
│  Для каждого kernel .c-файла:                              │
│    бросаем монету со смещением к GCC (скажем, 95/5)        │
│    → если GCC: компилировать через GCC (доверенный)        │
│    → если Claude: компилировать компилятором Claude        │
│                                                            │
│  Линкуем всё в образ ядра. Грузим его.                     │
│                                                            │
│  Если грузится: отлично. Берём большее подмножество Claude.│
│  Если ломается: подмножество Claude содержит баг.          │
│                 → бисекаем подмножество, сужаем,           │
│                   находим конкретный падающий файл,        │
│                   заводим сфокусированный баг-репорт,      │
│                   добавляем как изолированный падающий тест.│
└────────────────────────────────────────────────────────────┘
```

Псевдокод:

```python
def locate_bad_file(c_files: list[str]) -> str:
    # Стандартный бисек против известного-хорошего oracle.
    lo, hi = 0, len(c_files)
    while hi - lo > 1:
        mid = (lo + hi) // 2
        subset = c_files[lo:mid]
        image = build_kernel(subset, compiler="claude",
                             rest_compiler="gcc")
        if boots(image):
            lo = mid       # баг в верхней половине
        else:
            hi = mid       # баг в нижней половине
    return c_files[lo]
```

Как только вы локализовали баг в один файл, вы сделали критический шаг: превратили монолитный провал в **независимый, воспроизводимый тест-кейс**. Теперь вы снова в лёгком режиме — конкретный падающий тест, который может вести один Claude.

Общая форма этого паттерна:

> **Когда задача слишком велика для параллелизма, найдите доверенный oracle, прогоните бóльшую часть работы через oracle и позвольте агентам отгрызать сжимающееся подмножество, пока каждый остаточный провал не станет локальным.**

Это паттерн дизайна harness, а не компилятора. Он применим к чему угодно, где у вас монолитная система с доверенным референсом — data-пайплайны, компоненты обучения моделей, генерация конфигов, даже рендеринг UI.

## Специализация ролей агентов

Все 16 агентов — один и тот же бинарник Claude, но их промпты дают им разные *роли*. Звучит мелко; это не так. Наделение каждого агента отдельной ролью резко сокращает патологию «все столпились у одного провала».

Пять ролей из проекта, пересказ:

```
┌──────────────────────────┬──────────────────────────────────┐
│ Роль                     │ Вкус промпта                     │
├──────────────────────────┼──────────────────────────────────┤
│ duplicate-code-cleaner   │ Находить повторяющиеся паттерны, │
│                          │ извлекать хелперы, сокращать LOC │
│                          │ без изменения поведения.         │
├──────────────────────────┼──────────────────────────────────┤
│ performance-tuner        │ Профилировать, находить hot path,│
│                          │ применять локальные оптимизации. │
│                          │ Семантику не трогать.            │
├──────────────────────────┼──────────────────────────────────┤
│ codegen-optimizer        │ Фокус на бэкенд. Улучшать        │
│                          │ качество эмитируемого ассемблера. │
├──────────────────────────┼──────────────────────────────────┤
│ design-critic            │ Код не писать. Читать репо и     │
│                          │ заводить TODO на design smell,   │
│                          │ мёртвый код, несогласованность.  │
├──────────────────────────┼──────────────────────────────────┤
│ doc-maintainer           │ Держать README / PROGRESS-файлы  │
│                          │ честными и актуальными.          │
└──────────────────────────┴──────────────────────────────────┘
```

Разделение ролей важно, потому что оно **меняет, какие задачи агент считает «самыми очевидными».** `design-critic`, смотрящий на репо, видит другую «следующую работу», чем `performance-tuner` на том же репо. Флот покрывает больше поверхности работы без всякого центрального планировщика, решающего, кто что делает.

Для своего harness общее правило: **если два агента постоянно сталкиваются, обычно потому, что их промпты не различают, как выглядит «очевидный следующий шаг».**

## Когда НЕ строить agent-команду

Честная часть статьи. Agent-команды дороги, неудобны и изредка волшебны. Для большинства работ они не тот инструмент. НЕ стройте команду, если верно что-то из:

- **Проект мал.** Если одна сфокусированная Claude-session может закончить задачу за проход, флот из 16 даст оверхед координации, мерж-конфликты и счёт на $500 за работу на $5. Один острый нож бьёт шестнадцать тупых.
- **Работа требует плотной координации.** Дизайн с множеством сквозных решений — API-схемы, инварианты протоколов, разделяемые модели данных — параллельные агенты уничтожают. Агенты хороши в «сходи почини этот падающий тест». Плохи в «давайте договоримся, какой должна быть иерархия типов ошибок».
- **У вас нет хорошего test-oracle.** У компилятора есть GCC torture-test-сьют и ядро, которое либо грузится, либо нет. У большинства ПО нет доверенного, автоматизируемого, детерминированного сигнала «сработало?». Без него ваши агенты оптимизируют то, что проверяет верификатор, а не то, что вы реально хотите, — и поскольку верификатор неидеален, оптимизируют не то.
- **Вы не можете дёшево перезапуститься.** Ralph-loop'ы предполагают, что рестарт бесплатен. Если ваши агенты должны разогреваться 30 минут до полезной работы, цикл амортизируется плохо, и флот проигрывает одному долгоиграющему процессу.

Если что-то из этого про вас — возьмите одну Claude с хорошим промптом и остановитесь.

## Чему это учит harness-билдеров

Отходя от конкретного проекта, пять идей переносятся чисто:

1. **Git — примитив координации агентов.** Не стройте очередь задач, lock-сервис, шину сообщений. Используйте git. Локи, мержи, разрешение конфликтов, история, откат — всё получаете бесплатно, и это разработали люди, уже серьёзно думавшие о конкурентности.
2. **Качество тест-сьюта — потолок возможностей агента.** Claude хороша ровно настолько, насколько хорош сигнал, который вы даёте. Инвестируйте непропорционально в тесты, oracle'ы и верификаторы. Посредственный агент с отличным тест-сьютом бьёт отличного агента с посредственным сьютом каждый раз.
3. **Проектируйте harness для Claude, а не для себя.** Лаконичный дефолтный вывод. Маркеры `ERROR`. Полные логи в файлы. Детерминированное subsampling. Progress-файлы как письма будущим session. Всё это существует, потому что агент отличается от вас — забывчивее, легче отвлекается, слеп ко времени.
4. **Декомпозируйте монолиты через известные-хорошие oracle'ы.** Когда задача сопротивляется параллелизму, не добавляйте агентов; добавьте доверенный референс, прогоните через него основную массу работы и позвольте агентам атаковать сжимающийся остаток. GCC-as-oracle — общий паттерн.
5. **Специализируйте через промпты.** Одна модель, разные роли. Самый дешёвый и лучше всего масштабируемый способ сделать так, чтобы 16 агентов не наступали друг другу на ноги, — дать им разные определения «очевидного следующего шага».

Счёт на $20 000 оплатил компилятор. Бесплатным бонусом стал чертёж того, как agent-команды работают, когда они работают.
