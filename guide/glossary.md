# Глоссарий

Ключевые термины по всему гайду Harness Engineering, определённые простым языком.

---

**Harness (оболочка)**
Код, оборачивающий AI-модель и превращающий её в функционального агента. Управляет tool-loop'ом, context, memory и потоком решений. Без harness у вас чат-бот; с ним — агент.

**Runtime**
Среда выполнения, в которой работает harness. Включает ОС, файловую систему, сетевой доступ и sandbox. Runtime определяет, что агент физически *может сделать*, — harness определяет, что он *выбирает* сделать.

**Framework (фреймворк)**
Библиотека или SDK для сборки harness'ей. Фреймворки дают абстракции (реестры tools, memory-хранилища, шаблоны промптов), чтобы не строить с нуля. Примеры: LangChain, CrewAI, AutoGen. Фреймворк — инструментарий; harness — готовый продукт.

**Context Window (окно контекста)**
Максимум текста (в токенах), который модель может обработать за один запрос. Обычно 128K–200K токенов для fronttier-моделей. Всё, что агент «знает» за ход, — инструкции, memory, история диалога, содержимое файлов — должно уложиться в это окно.

**Tool Loop (цикл инструментов)**
Базовый цикл выполнения агента: вызвать модель → модель запрашивает tool → выполнить tool → вернуть результат → повторить. Цикл идёт, пока модель не выдаст финальный текстовый ответ без tool-вызовов. Любой harness реализует какую-то версию этого цикла.

**Skill (скилл)**
Самодостаточный пакет возможности, расширяющий способности агента. Состоит из файла-инструкции SKILL.md плюс любой сопутствующий код или шаблоны. Skills — «толстая» часть паттерна «тонкий harness, толстые skills»: они кодируют доменное знание, оставляя harness универсальным.

**AGENTS.md**
Конфиг-файл, определяющий, как агент должен вести себя в workspace. Содержит инструкции, конвенции, рекомендации по использованию tools и определения workflow. Читается harness'ем в начале каждой session. Портативен между реализациями harness, поддерживающими конвенцию.

**MEMORY.md**
Файловое хранилище долгосрочной memory. Содержит курируемое знание, нужное агенту между сессиями: предпочтения пользователя, детали проекта, выученные уроки. В отличие от проприетарных memory-систем, MEMORY.md читаем, редактируем и портируем человеком.

**ReAct (Reasoning + Acting)**
Промпт-паттерн, где модель чередует мышление («мне нужно поискать X») и действие (вызов tool). Даёт цикл Thought → Action → Observation. Большинство современных agent-harness реализуют вариант ReAct, часто неявно через tool-loop.

**Chain-of-Thought (CoT)**
Промпт-техника, где модель поощряют «думать шаг за шагом» перед финальным ответом. Extended thinking (как thinking-режим Claude) — модельная реализация этого: модель генерирует внутренние reasoning-токены перед ответом.

**HaaS (Harness as a Service)**
Паттерн развёртывания agent-harness как управляемых облачных сервисов. API-gateway маршрутизирует запросы в session-менеджеры, которые диспетчеризуют в harness-воркеры в sandboxed-средах. Включает multi-tenant-хостинг агентов с изоляцией session и лимитами ресурсов.

**Thin Harness (тонкий harness)**
Философия дизайна, где сам harness минимален — только tool-loop, управление context и оркестрация session, — а доменная логика живёт в skills. harness — универсальный движок; skills дают экспертизу. Противоположность монолитным agent-архитектурам.

**Thick Skills (толстые skills)**
Дополнение «тонкому harness'у». Skills содержат детальные инструкции, примеры и workflow под конкретные задачи. Толстый skill для почты может включать шаблоны, тональные гайдлайны и платформо-специфичные правила форматирования. Skill — там, где живёт доменное знание.

**Multi-Agent (мультиагентная архитектура)**
Архитектура, где несколько AI-агентов коллаборируют над задачей. Агенты могут иметь разные роли (researcher, writer, reviewer), разные модели или разные наборы tools. Требует оркестрации — решения, кто что делает и как они общаются.

**Sub-Agent (субагент)**
Агент, порождённый другим агентом для конкретной подзадачи. Родительский агент делегирует работу и получает результаты по завершении sub-agent'а. Sub-agent'ы имеют изолированные session, но могут разделять memory или доступ к workspace. Ключевой паттерн декомпозиции сложных задач.

**Sandbox (песочница)**
Изолированная среда выполнения для безопасного запуска tool-вызовов агента. Обычно контейнер или VM с ограниченной файловой системой, сетью и доступом к процессам. Предотвращает влияние агента на хост-систему (случайное или злонамеренное). Необходим для tools, выполняющих код.

**Context Compression (сжатие контекста)**
Техники размещения большего количества информации в фиксированном context window. Включают саммаризацию старых сообщений, удаление избыточного контента, усечение крупных tool-выводов и селективную загрузку только релевантных файлов. Критично, когда история диалога превышает лимит context.

**Token Budget (токен-бюджет)**
Стратегия распределения context window под разные цели: system-промпт, memory, история диалога, результаты tools и вывод модели. Хороший токен-бюджет гарантирует приоритет самой важной информации. Пример: резерв 20% под system-промпт, 15% под memory, 50% под историю, 15% под ответ.

**Session (сессия)**
Одно непрерывное взаимодействие между пользователем и агентом. У session есть начало (юзер шлёт первое сообщение), середина (back-and-forth с tool-вызовами) и конец (юзер выходит или session таймаутится). Состояние session включает историю диалога, активные tools и накопленный context.

**Checkpoint (чекпоинт)**
Сохранённый слепок состояния агента в конкретной точке session. Включает историю диалога, состояния tools и промежуточные результаты. Позволяет возобновиться после падений или пауз без рестарта с нуля. Критично для долгоиграющих задач, где провал на середине дорог.

**MCP (Model Context Protocol)**
Открытый протокол подключения AI-моделей к внешним tools и источникам данных. Определяет стандартный интерфейс, чтобы tools (MCP-серверы) могли использоваться любым совместимым harness'ем (MCP-клиентом). Решает N×M-проблему интеграции: вместо того чтобы каждый harness интегрировался с каждым tool, обе стороны говорят на MCP.

**Eval (оценка)**
Процесс измерения производительности агента по заданным критериям. Включает поведенческие тесты (вызвал ли агент правильные tools?), тесты исхода (правильный ли результат?) и экономические тесты (сколько стоило?). Eval'ы для агентов — то же, что unit-тесты для кода: в production не опциональны.

**Trace (трейс)**
Полная запись agent-session: каждое отправленное сообщение, каждый вызванный tool, каждый полученный результат и финальный вывод. Трейсы используются для дебага (что пошло не так?), оценки (как отработал агент?) и регрессионного тестирования (новая версия хуже?). Agent-аналог лог-файла.
