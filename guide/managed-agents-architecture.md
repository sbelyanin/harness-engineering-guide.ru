# Архитектура Managed Agents: отвязываем Brain от Hands

> **Главный инсайт:** Монолитный agent-контейнер — где harness, sandbox и состояние session делят один процесс — это pet, которого вы не можете потерять. Отвязка brain (harness + LLM) от hands (sandbox + tools) и session (event-лог) превращает каждый компонент в заменяемое cattle и открывает независимое масштабирование, восстановление после сбоев и изоляцию безопасности.

## Проблема монолита

Простейшая agent-архитектура кладёт всё в один box:

```
┌──────────────────────────────────┐
│         Один контейнер            │
│                                   │
│  ┌─────────┐  ┌──────────────┐   │
│  │ Harness  │  │   Sandbox    │   │
│  │ (brain)  │  │ (exec кода)  │   │
│  │          │  │              │   │
│  │ Session  │  │ Креды        │   │
│  │ state    │  │ Файлы юзера  │   │
│  └─────────┘  └──────────────┘   │
│                                   │
└──────────────────────────────────┘
```

Это работает для прототипов. Для production падает тремя предсказуемыми способами.

### Провал 1: Проблема Pet

В аналогии [pets-vs-cattle](https://cloudscaling.com/blog/cloud-computing/the-history-of-pets-vs-cattle/) pet — это именный, ручной сервер, который нельзя терять. Cattle — взаимозаменяемы: когда одно умирает, его заменяют.

Монолитный agent-контейнер — pet. Если контейнер падает, session теряется. Если зависает — приходится его «лечить». Просто убить и поднять новый нельзя, потому что состояние session — полная история того, что агент сделал, — живёт внутри.

### Провал 2: Слепое пятно при дебаге

Когда монолитный контейнер висит, единственное окно — внешний event-stream. Но баг harness, дроп пакета и упавший контейнер снаружи выглядят одинаково. Чтобы диагностировать реальную причину, инженеру нужен shell внутри контейнера — но там же лежат данные пользователя и креды. Дебаг превращается в security-инцидент.

### Провал 3: Граница безопасности

Недоверенный код, который генерирует агент, выполняется в том же контейнере, что и креды. Prompt injection достаточно убедить агента прочитать собственные переменные окружения. Получив токены, атакующий может поднимать неограниченные сессии. Узкое скопирование помогает, но кодирует предположения о том, что модель не сможет сделать с ограниченными токенами — а модели всё умнее.

## Трёхслойная архитектура

Фикс — разложить монолит на три независимых слоя, каждый со своим lifecycle:

```
┌─────────────────────────────────────────────────────────┐
│                    Слой оркестрации                       │
│                                                          │
│  ┌──────────────┐  ┌──────────┐  ┌───────────────────┐  │
│  │    Brain      │  │ Session  │  │      Hands        │  │
│  │  (Harness +   │  │ (Event-  │  │  (Sandbox A)      │  │
│  │   LLM-вызовы) │  │  лог)    │  │  (Sandbox B)      │  │
│  │              │  │          │  │  (MCP Tools)      │  │
│  │  Stateless   │  │ Durable  │  │  Disposable       │  │
│  └──────┬───────┘  └────┬─────┘  └────────┬──────────┘  │
│         │               │                  │             │
│         │  emitEvent()  │   execute()      │             │
│         ├──────────────►│◄─────────────────┤             │
│         │  getEvents()  │   provision()    │             │
│         ├──────────────►│                  │             │
│         └───────────────┴──────────────────┘             │
└─────────────────────────────────────────────────────────┘
```

### Brain (Harness)

harness крутит agentic loop: вызывает LLM, маршрутизирует tool-вызовы, управляет context. Что критично — harness **stateless**. Он не держит данные session в памяти. Всё долговременное идёт в лог session через `emitEvent()`.

Когда harness падает, поднимается новый, вызывает `wake(sessionId)`, подтягивает event-лог через `getEvents()` и возобновляется с последнего события. Без потери данных. Без «лечения». harness стал cattle.

### Session (Event-лог)

Session — **append-only лог** всего, что произошло: LLM-вызовы, результаты tools, сообщения пользователя, системные события. Живёт вне и harness, и sandbox — в долговременном хранилище.

Ключевые интерфейсы:
- `emitEvent(sessionId, event)` — записать событие в ходе agent-loop
- `getEvents(sessionId, options)` — прочитать события (позиционное срезание, фильтрация)
- `getSession(sessionId)` — получить метаданные и статус

Session переживает и harness, и sandbox. Если что-то из них падает, session остаётся нетронутой.

### Hands (Sandbox)

Sandbox — среды выполнения, где агент запускает код, правит файлы и выполняет команды. Создаются по требованию через `provision({resources})` и уничтожаются, когда больше не нужны.

harness вызывает sandbox так же, как любой tool: `execute(name, input) → string`. Если sandbox умирает, harness ловит ошибку как провальный tool-вызов и передаёт её LLM. Модель может решить ретраить на свежем sandbox.

## Session ≠ Context Window

Это различие — самая тонкая и самая важная часть архитектуры.

**Session** — полная, долговременная запись всего: потенциально миллионы токенов за часы или дни работы агента. **Context window** — подмножество этой записи, которое harness отбирает для текущего LLM-вызова — обычно 128K-200K токенов.

```
Session (append-only event-лог, durable)
┌─────────────────────────────────────────────────────────────┐
│ event_1 │ event_2 │ ... │ event_500 │ ... │ event_2000     │
└─────────────────────────────────────────────────────────────┘
                                    │
                          getEvents(slice)
                                    │
                                    ▼
                    Context Window (выбранное подмножество)
                    ┌───────────────────────────┐
                    │ system_prompt             │
                    │ event_1950 ... event_2000 │
                    │ (50 самых свежих событий) │
                    └───────────────────────────┘
```

У этого разделения три плюса:

**1. Необратимые решения становятся обратимыми.** Compaction (саммаризация старого context ради места) навсегда уничтожает информацию. С долговременным логом session harness может перечитать любое прошлое событие — даже то, что ранее было сжато из context window. Если будущему ходу нужна информация из event_200, harness её подтянет.

**2. Context engineering становится задачей harness.** harness решает, что попадает в context window: недавние события, саммари старых, system-промпты, релевантное содержимое файлов. Разные реализации harness могут использовать разные стратегии, не меняя формат session. Сегодня context engineering — токен-level-обрезка, завтра — семантический retrieval. Session остаётся прежней.

**3. Multi-brain-архитектуры становятся возможными.** Несколько harness могут одновременно читать один лог session. Planning-brain может ревьюить полную историю, пока execution-brain фокусируется на недавних событиях. Оба читают из одного durable-источника.

## Безопасность: креды никогда не входят в Sandbox

Развязанная архитектура создаёт естественную границу безопасности. Креды живут в vault вне sandbox. Агент никогда не держит токены напрямую.

Два паттерна:

**Вшитые в ресурс:** для Git access-токен используется при инициализации sandbox, чтобы клонировать репо и настроить локальный remote. `git push` и `git pull` работают изнутри sandbox, и агент никогда не видит токен.

**Vault + proxy:** для внешних API (через MCP) OAuth-токены живут в защищённом vault. Агент вызывает MCP-tools через proxy. proxy поднимает креды session из vault и делает внешний вызов. sandbox никогда не видит креды.

```
┌───────────┐     ┌───────────┐     ┌───────────┐     ┌──────────┐
│  Agent    │────►│  MCP      │────►│  Vault    │────►│ External │
│ (sandbox) │     │  Proxy    │     │ (секреты) │     │ Service  │
│           │     │           │     │           │     │          │
│ Кред нет  │     │ Session   │     │ OAuth-    │     │          │
│           │     │ → креды   │     │ токены    │     │          │
└───────────┘     └───────────┘     └───────────┘     └──────────┘
```

Даже если prompt injection убедит агента обыскать всё окружение, кред там искать нечего.

## Производительность: улучшения TTFT

В монолитном дизайне каждая session — даже та, что не трогает sandbox, — платила полную стоимость настройки контейнера: клон репо, бут процесса, подтягивание событий. Inference не мог стартовать, пока контейнер не готов.

В отвязанной архитектуре harness стартует немедленно. Provisioning sandbox происходит лениво через tool-вызов, только если нужно. Результаты production-развёртывания Anthropic:

| Метрика | Монолит | Отвязанная | Улучшение |
|--------|-----------|-----------|-------------|
| TTFT p50 | baseline | ~40% от baseline | **~60% сокращение** |
| TTFT p95 | baseline | ~10% от baseline | **~90% сокращение** |

Улучшение p95 драматично, потому что худший сценарий старта контейнера (холодные pull, медленный диск) полностью убран из критического пути для сессий, которым sandbox не нужен.

## Много Brain, много Hands

Архитектура естественно расширяется в обоих измерениях.

### Масштабирование Brain

Масштабирование до множества concurrent session агентов означает запуск множества stateless-harness. Поскольку каждый harness не держит состояния (всё долговременное — в логе session), масштабирование горизонтальное и тривиальное. Никакой разделяемой памяти, распределённых локов, синхронизации состояния.

### Масштабирование Hands

Один brain может подключаться к нескольким sandbox. На практике это значит, что агент рассуждает о нескольких средах выполнения и решает, куда отправить работу:

```
┌──────────┐
│  Brain   │
│ (harness)│
└────┬─────┘
     │
     ├──► Sandbox A (Python-окружение)
     ├──► Sandbox B (Node.js-окружение)
     ├──► Sandbox C (VPC клиента)
     └──► MCP Tool Server (внешние API)
```

Ранние модели с этим не справлялись — рассуждение о нескольких средах одновременно было слишком сложно. По мере роста возможностей модели допущение одного контейнера стало узким местом. Отвязанная архитектура была готова к тому, что модели догонят.

### Интеграция с VPC клиента

Когда всё работало в одном контейнере, подключение к инфраструктуре клиента означало network peering — VPC клиента должен был быть доступен из нашего контейнера. В отвязанной архитектуре sandbox может работать внутри VPC клиента, пока brain крутится в нашей инфраструктуре. brain общается с sandbox через стандартный интерфейс `execute()`, независимо от того, где физически работает sandbox.

## Антипаттерны

### Хранение состояния Session в Harness

```python
# ❌ Session state in harness memory
class HarnessWithState:
    def __init__(self):
        self.events = []  # Lost if harness crashes
        self.current_plan = None
    
    def run_turn(self, message):
        result = self.llm.call(self.events + [message])
        self.events.append(result)  # Only in memory
```

Если harness падает, всё пропадает. Всегда эмитьте события в долговременный лог session.

```python
# ✅ Session state externalized
class StatelessHarness:
    def run_turn(self, session_id, message):
        events = self.session_store.get_events(session_id)
        result = self.llm.call(events + [message])
        self.session_store.emit(session_id, result)  # Durable
```

### Креды в Sandbox

```python
# ❌ Passing API keys into the sandbox environment
sandbox.execute("export GITHUB_TOKEN=ghp_xxx && git push")
```

В одном prompt injection от экзфильтрации. Используйте паттерн vault + proxy.

### Жадный provisioning Sandbox

```python
# ❌ Provision sandbox before knowing if it's needed
sandbox = provision_sandbox(resources=large_config)  # 30s startup
result = llm.call(prompt)  # Might not even need the sandbox
```

Provision'ьте лениво. Пусть LLM через tool-вызов решит, нужен ли sandbox и какой.

## Чек-лист проектирования

Перед сборкой managed-agent-архитектуры:

1. **Stateless brain** — может ли harness упасть и перезапуститься без потери данных?
2. **Durable session** — хранится ли event-лог вне и brain, и hands?
3. **Disposable sandbox** — можно ли любую sandbox убить и заменить?
4. **Изоляция кредов** — недоступны ли секреты из sandbox?
5. **Ленивый provisioning** — создаются ли sandbox по требованию, а не заранее?
6. **Стабильность интерфейсов** — достаточно ли общие интерфейсы brain↔hands, чтобы пережить текущие реализации?

## Резюме

Монолитный agent-контейнер — естественная стартовая точка: всё в одном box просто. Но он создаёт pet'ов, которых нельзя заменить, границы безопасности, которые нельзя обеспечить, и узкие места производительности, которые нельзя убрать.

Разложение на три слоя — stateless brain, durable session, disposable hands — решает все три. brain становится cattle (упал и перезапустился). session становится единым источником правды (переживает всё остальное). hands становятся ресурсами по требованию (provision только когда нужно, destroy когда сделано).

Это тот же паттерн, который операционные системы открыли десятилетия назад: виртуализуйте компоненты, делайте интерфейсы стабильными и позволяйте реализациям свободно меняться под ними. Syscall `read()` не знает, читает ли он дисковый пакет 1970-х или современный SSD. Интерфейс `execute()` не знает, работает ли sandbox локально, в VPC клиента или на другом континенте.
