# Битва за миллиард токенов: релиз нашего Windows-клиента OpenClaw

> **Nexu** — open-source, one-click-install десктопный клиент OpenClaw, который даёт AI в ваши руки — локально.
> GitHub: **https://github.com/nexu-io/nexu**
> ⭐ Если находите проект полезным, Star для нас много значит.

На прошлой неделе мы выпустили Nexu v0.1.12 с полной поддержкой Windows. Будучи первым в мире open-source десктопным клиентом OpenClaw — обрабатывающим **миллиарды токенов ежедневно** — Windows был самым запрашиваемым сообществом функционалом.

Но "Windows support" — три слова, которые скрывают полную перестройку packaging pipeline. Вот что изменилось:

| Метрика | Было | Стало |
|--------|------|-------|
| ⏱ **Время сборки** | 15 мин | **4 мин** |
| 📦 **Время установки** | 10 мин | **менее 2 мин** |
| 🔄 **CI-сборки** | ручные | **полностью автоматизированы** |
| 🧩 **Логика обновлений** | связанная | **macOS/Windows декуплингованы** |

По пути мы уперлись во множество стен и приняли несколько нетривиальных технических решений. Раз мы open source, **нет причин держать эти уроки при себе**. Это первый пост из серии — надеемся, он сэкономит вам время, если вы поставляете Electron-приложения кросс-платформенно.

## Где electron-builder не дотягивает

![Where electron-builder Falls Short](https://raw.githubusercontent.com/joeylee12629-star/article-assets/main/win-packaging/sec-en-01.jpg)

electron-builder — де-факто стандарт для упаковки Electron-приложений. Большинство проектов используют его end-to-end — от исходников до финального инсталлятора. Так делали и мы, изначально.

Но как только мы запустили полный Windows pipeline в production, стандартный путь начал разваливаться:

- Дерево упакованных файлов Nexu содержит примерно **38 000 файлов**. Стандартное ZIP-сжатие ужасно справляется с таким масштабом — и по времени сборки, и по времени установки
- Нам нужна была **кастомная логика установщика** — опции миграции данных, очистка реестра, пользовательские выборы, которые дефолты electron-builder не поддерживают
- macOS и Windows имеют **фундаментально разную семантику обновлений**. Запихивать их в один кодовый путь создавало хрупкое, трудноотлаживаемое поведение
- CI-артефакты должны быть **репродуцируемы** независимо от настройки локальной машины

electron-builder всё ещё отвечает за производство `win-unpacked` (сырой каталог приложения). Но с этого момента мы взяли управление на себя.

## Кастомный pipeline: 7z + NSIS

![Custom 7z + NSIS Pipeline](https://raw.githubusercontent.com/joeylee12629-star/article-assets/main/win-packaging/sec-en-02.jpg)

Наш новый pipeline состоит из двух стадий:

**Стадия 1: electron-builder производит win-unpacked**
Стандартный каталог Electron-приложения — exe, dll, ресурсы. electron-builder всё ещё хорошо справляется с этим.

**Стадия 2: Кастомная упаковка берёт управление**

- Вендоризированный **7-Zip** сжимает win-unpacked в `payload.7z`
- **makensis** компилирует кастомный NSIS-инсталлятор
- Инсталлятор отвечает за распаковку, записи в реестр, ярлыки и очистку при удалении

**Почему 7z?** С 38 000 файлами solid-сжатие 7z обрабатывает всё дерево как один блок — драматически лучшее соотношение сжатия и скорость распаковки, чем у ZIP. Это главная причина, по которой время установки упало с 10 минут до менее 2.

**Почему NSIS?** Полный контроль над каждым аспектом флоу установки — пути, опции миграции, реестр, удаление. Больше не нужно бороться с дефолтами фреймворка.

Мы также вендоризировали 7-Zip прямо в репо, так что и CI, и локальные сборки имеют ноль внешних зависимостей. Репродуцируемость из коробки.

## Платформенные Update Driver'ы

![Platform Update Drivers](https://raw.githubusercontent.com/joeylee12629-star/article-assets/main/win-packaging/sec-en-03.jpg)

Этот момент укусил нас сильно. Наша логика обновлений для macOS и Windows раньше использовала один кодовый путь, разветвлённый через `if (platform === 'win32')`.

Проблема в том, что у этих платформ фундаментально разные модели обновлений:

- macOS может **тихо обновляться внутри приложения**
- Windows требует **quit → run installer → restart**

Запихивание обеих в один путь означало, что каждый фикс под Windows рисковал сломать macOS. Поэтому мы ввели абстракцию **Update Driver** — три независимых платформенных драйвера:

- **mac-update-driver** — in-app download и установка
- **windows-update-driver** — внешний download + redirect на инсталлятор
- **unsupported-update-driver** — graceful fallback для других платформ

Каждая платформа теперь развивается независимо. Если вы строите кросс-платформенные Electron-приложения, **введите это разделение пораньше** — потом скажете себе спасибо.

## Разрешение runtime-путей

![Runtime Path Resolution](https://raw.githubusercontent.com/joeylee12629-star/article-assets/main/win-packaging/sec-en-04.jpg)

Вот тонкий момент: после упаковки runtime-расположения файлов зависят от конфига сборки. Наш старый код использовал свободные эвристики для их поиска — что обычно работало, пока однажды не перестало.

Любое изменение в layout вывода сборки могло тихо сломать runtime-обнаружение. И эти падения проявлялись только в упакованных сборках — никогда в локальной dev-среде.

Мы написали выделенный Windows runtime locator, который разрешает пути **явно относительно exe**. Контракт между layout вывода сборки, runtime-упаковкой и runtime-обнаружением теперь жёсткий.

До: "наверное, где-то рядом". Теперь: **"должно быть точно здесь, иначе громко падаем"**. Больше никакого тихого разрушения в production.

## CI/CD Pipeline

![CI/CD Pipeline](https://raw.githubusercontent.com/joeylee12629-star/article-assets/main/win-packaging/sec-en-05.jpg)

Раньше Windows-сборки были исключительно локальной историей — успех зависел от правильной настройки тулчейна на вашей машине.

Мы перенесли весь флоу build → package → sign → publish в GitHub Actions:

- **Nightly / beta / release** — все три workflow теперь полностью поддерживают Windows
- Автогенерация `latest-win.json` манифестов обновлений
- Артефакты включают инсталлятор, хеши и метаданные — полностью прослеживаемы

Для open-source проектов репродуцируемый CI — всё. Любой контрибьютор может собрать и получить точно такой же вывод.

## Что дальше

![What's Next](https://raw.githubusercontent.com/joeylee12629-star/article-assets/main/win-packaging/sec-en-06.jpg)

Реальная ценность этой работы не в красивой новой фиче — а в **инфраструктуре, которая накапливает эффект**. Больше контроля над упаковкой, более стабильные обновления, более предсказуемые пути.

У нас в pipeline ещё есть экспериментальные оптимизации (например, lifecycle-driven миграция Windows-данных), которые мы рассмотрим в следующем посте, как только будут battle-tested.

Весь смысл open source — **делиться тем, что вы узнали, а не только тем, что вы выпустили**.
