# Multi-Agent Orchestration

> **Главный инсайт:** Один агент упирается в жёсткие лимиты — размер context window, доменную специализацию и последовательное выполнение. Multi-agent-оркестрация разбивает работу между несколькими агентами, каждый со своим context и tools, координируемыми паттернами, заимствованными из распределённых систем.

Статья [Sub-Agent](sub-agent.md) разбирает простейший случай: один лидер порождает воркеров для параллельных задач. Эта статья идёт дальше — паттерны оркестрации для систем, где несколько агентов коллаборируют, специализируются и масштабируются.

## Зачем Multi-Agent?

Один агент с одним Agentic Loop — это дефолт. Он хорошо работает для большинства задач. Но три стены в конечном счёте заставляют выйти за его пределы:

### 1. Лимиты Context Window

Даже с окнами 200K токенов сложный проект может исчерпать доступный context. Рефакторинг кодовой базы по 80 файлам, исследование с 40 источниками, многостадийный pipeline с накоплением tool-выводов — всё это может переполнить одно окно. Когда context заполняется, модель начинает забывать ранние инструкции и терять связность.

Multi-agent делит работу. Каждый агент получает свежий context window под свой срез задачи.

### 2. Специализация

Генералист, который пишет код, ревьюит PR, генерирует маркетинговый текст и гоняет анализ данных, посредствен во всём. Агенты работают лучше, когда их system-промпт, tools и context заточены под конкретный домен. «Code-review-агент» с загруженными lint-инструментами и style-guide выдаёт более качественные ревью, чем general-purpose-агент, у которого просто есть доступ к линтеру.

### 3. Параллелизм

Один агент последователен. Вызывает один tool, ждёт результат, рассуждает, вызывает другой. Когда у вас пять независимых задач — тесты для пяти модулей, исследование пяти конкурентов, перевод на пять языков — один агент обрабатывает их по очереди. Пять агентов — одновременно.

## Паттерны оркестрации

Четыре паттерна покрывают большинство multi-agent-архитектур. Они не взаимоисключающие — production-системы часто их комбинируют.

### Последовательный Pipeline

Агент A заканчивает работу, передаёт вывод Агенту B, который передаёт свой вывод Агенту C. Каждый агент трансформирует или обогащает результат.

```
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
 │ Ingest  │────►│ Analyze │────►│  Draft  │────►│ Review  │
 │  Agent  │     │  Agent  │     │  Agent  │     │  Agent  │
 └─────────┘     └─────────┘     └─────────┘     └─────────┘
```

**Когда использовать:** задачи с естественными стадиями, где каждая требует другой экспертизы или tools. Пример: собрать данные → почистить и проанализировать → сгенерировать отчёт → вычитать и проверить факты.

**Ключевое свойство:** каждый агент видит только вывод предыдущей стадии, а не весь накопленный context. Это фича — она принуждает к чётким интерфейсам между стадиями и предотвращает раздувание context.

### Fan-Out / Fan-In

Координатор диспетчеризует одинаковые (или связанные) задачи N агентам параллельно, затем собирает и мержит все результаты.

```
                    ┌───────────┐
               ┌───►│ Worker A  │───┐
               │    └───────────┘   │
┌──────────┐   │    ┌───────────┐   │    ┌──────────┐
│Dispatcher│───┼───►│ Worker B  │───┼───►│  Merger  │
└──────────┘   │    └───────────┘   │    └──────────┘
               │    ┌───────────┐   │
               └───►│ Worker C  │───┘
                    └───────────┘
```

**Когда использовать:** задачи, декомпозируемые на независимые, параллелизуемые подзадачи. Пример: перевести документ на 8 языков, написать unit-тесты для 10 модулей, исследовать 5 продуктов-конкурентов.

**Ключевое свойство:** воркеры не взаимодействуют друг с другом. Merger-агент (или сам диспетчер) отвечает за комбинирование результатов и разрешение конфликтов.

### Supervisor

Один супервизорный агент принимает все решения о делегировании. Он наблюдает текущее состояние, решает, какого специалиста вызвать следующим, читает результат и решает следующий шаг. В отличие от pipeline, supervisor может циклиться, переназначать или вызывать одного специалиста многократно по промежуточным результатам.

```
                 ┌─────────────────┐
                 │   Supervisor    │
                 │ (решает, цикл)  │
                 └───┬───┬───┬────┘
                     │   │   │
            ┌────────┘   │   └────────┐
            ▼            ▼            ▼
      ┌──────────┐ ┌──────────┐ ┌──────────┐
      │ Code     │ │ Research │ │ Review   │
      │ Agent    │ │ Agent    │ │ Agent    │
      └──────────┘ └──────────┘ └──────────┘
```

**Когда использовать:** задачи, где следующий шаг зависит от результата предыдущего, а решение, *какого* специалиста вызвать, требует суждения. Пример: coding-supervisor, делегирующий реализацию code-агенту, отправляющий результат review-агенту и возвращающийся к code-агенту, если ревью нашло проблемы.

**Ключевое свойство:** supervisor держит общий план и состояние. Специалисты stateless — получают задачу, возвращают результат, забывают. Так context window специалистов остаются чистыми.

### Peer-to-Peer

Агенты общаются напрямую без центрального координатора. У каждого свой inbox, и каждый может слать сообщения любому другому.

**Когда использовать:** симуляции, рассуждения в стиле дебатов (агент A за, агент B против, агент C судит) или системы, где агенты представляют разных стейкхолдеров.

**Ключевое свойство:** нет единой точки координации. Это делает peer-to-peer самым гибким паттерном, но и самым сложным в дебаге. Без supervisor трудно гарантировать завершение или отслеживать прогресс.

**Предупреждение:** peer-to-peer редко правильный выбор в production-harness-системах. Отсутствие центрального контроля сильно усложняет обработку ошибок, enforcement таймаутов и сбор результатов. Предпочитайте Supervisor или Fan-Out/Fan-In, если нет специфической причины для децентрализованного общения.

## Реализация в Harness

harness (хост-программа, оборачивающая LLM в рабочего агента — см. [Что такое Harness](what-is-harness.md)) реализует multi-agent-оркестрацию через четыре механизма: порождение sub-agent'ов, изоляция context, общение «родитель читает ребёнка» и обработку таймаутов.

### Порождение Sub-Agent

Фундаментальный примитив — порождение изолированной agent-session. В OpenClaw это паттерн `sessions_spawn`:

```
Parent Agent
    │
    ├── sessions_spawn(label="research", task="Find competitor pricing")
    │       → создаёт изолированную session со своим context, tools, моделью
    │
    ├── sessions_spawn(label="writer", task="Draft the pricing page")
    │       → ещё одна изолированная session
    │
    └── ждёт завершения обеих (push-based)
```

Каждая порождённая session — полностью независимый агент со своим:
- Context window (без разделяемого токен-бюджета с родителем)
- System-промптом (можно кастомизировать под sub-agent'а)
- Набором tools (можно ограничить или расширить)
- Моделью (можно использовать дешевле для простых задач)

Родитель указывает label, описание задачи и опционально модель или уровень thinking. harness берёт на себя lifecycle процесса, таймаут и доставку результата.

### Изоляция Context

Изоляция context — не удобство, а ключевое проектное ограничение. Context window каждого sub-agent'а полностью независим:

```
┌──────────────────────┐    ┌──────────────────────┐
│   Parent Context     │    │  Sub-Agent Context    │
│                      │    │                       │
│ [system prompt]      │    │ [system prompt]       │
│ [диалог юзера]       │    │ [инструкция задачи]   │
│ [результаты A,B,C]   │    │ [результаты X,Y]      │
│ [128K токенов]       │    │ [12K токенов]         │
│                      │    │                       │
│  Не видит ──────────►│ X  │◄──── Не видит         │
│  context sub-agent'а │    │  context родителя     │
└──────────────────────┘    └──────────────────────┘
```

**Почему это важно:** родитель с 128K токенов накопленного диалога может породить sub-agent'а, который стартует с чистым бюджетом 200K. Sub-agent не отягощён нерелевантной историей — он видит только инструкцию задачи и тот context, который родитель явно передал.

Следствие: sub-agent'ы не могут читать переменные родителя, результаты его tools или историю диалога. Вся передача информации явная — родитель включает релевантный context в строку инструкции задачи.

### Общение: родитель читает результаты ребёнка

Multi-agent-общение в harness следует строгому паттерну: **родитель читает результаты ребёнка; дети не пишут в memory родителя.**

```
1. Родитель порождает ребёнка с инструкцией задачи (однонаправленная запись)
2. Ребёнок выполняется независимо
3. Ребёнок завершается → harness доставляет результат родителю (push-based)
4. Родитель читает результат в следующей итерации Agentic Loop
```

Нет разделяемой memory, нет шины сообщений, нет callback, где ребёнок меняет состояние родителя. Это ограничение резко упрощает рассуждения о системе — родитель всегда единственный источник правды об общем прогрессе.

В OpenClaw завершение **push-based**: когда sub-agent заканчивает, результат автоматически анонсируется в session родителя. Родителю не нужно поллить. Это устраняет самый частый multi-agent-баг: плотные polling-циклы, сжигающие токены на проверку «готово ли?».

### Обработка таймаутов и ошибок

Sub-agent'ы могут зависать, падать или выдавать мусор. Production-harness должен обработать все три случая:

```python
@dataclass
class SubAgentConfig:
    task: str
    label: str
    timeout_seconds: int = 300       # Kill after 5 minutes
    max_retries: int = 1             # Retry once on failure
    fallback: str | None = None      # Alternative task on repeated failure

class SubAgentManager:
    def spawn_with_safeguards(self, config: SubAgentConfig) -> str:
        for attempt in range(config.max_retries + 1):
            try:
                result = self.spawn(
                    task=config.task,
                    label=config.label,
                    timeout=config.timeout_seconds,
                )
                if self.validate_result(result):
                    return result
                # Invalid result — retry with clarified instructions
                config.task += "\n\nPrevious attempt produced invalid output. Be precise."
            except TimeoutError:
                if attempt == config.max_retries and config.fallback:
                    return self.spawn(task=config.fallback, label=f"{config.label}-fallback")
                continue
            except Exception as e:
                log.error(f"Sub-agent {config.label} failed: {e}")
                if attempt == config.max_retries:
                    return f"[FAILED] {config.label}: {e}"
        return f"[FAILED] {config.label}: max retries exceeded"
```

**Ключевые принципы:**
- **Всегда ставьте таймауты.** Sub-agent без таймаута — ждущая случиться утечка ресурсов.
- **Валидируйте результаты.** Sub-agent, «завершившийся» пустым выводом или сообщением об ошибке, — не успех.
- **Деградируйте изящно.** При провале sub-agent'а родитель должен продолжать с частичными результатами, а не абортить — если только провалившаяся задача не критична.
- **Ограничивайте глубину.** Sub-agent'ы, порождающие своих sub-agent'ов, создают экспоненциальную сложность. Капайте рекурсию на 1–2 уровня.

## Реальные примеры

Три open-source-проекта демонстрируют multi-agent-оркестрацию в разных масштабах.

### Multica (14K+ звёзд)

**Паттерн:** Supervisor + Fan-Out / Fan-In

Multica трактует агентов как тиммейтов на project-борде. Человек (или lead-агент) назначает GitHub-issues agent-воркерам, каждый работает независимо. Система отслеживает прогресс на Kanban-подобном борде, показывая, кто над чем работает, что в ревью, что смержено.

Ключевые проектные решения:
- **Поддержка multi-runtime** — агенты могут работать на Claude Code, Codex, OpenClaw или OpenCode. Оркестратору всё равно, какой runtime у воркера — лишь бы выдавал ожидаемый вывод.
- **Составные skills** — агенты могут комбинировать несколько Skills (например, агент «fix-and-test» грузит и coding-Skill, и testing-Skill).
- **Issue-driven-делегирование** — работа организована вокруг GitHub-issues, а не абстрактных «задач». Это даёт каждому агенту конкретный, отслеживаемый unit работы с чёткими критериями приемки.

### Paseo (3.6K+ звёзд)

**Паттерн:** Supervisor с кросс-устройственным выполнением

Paseo даёт единый интерфейс для взаимодействия с несколькими агентами. Вы даёте команду, Paseo маршрутизирует её подходящему агенту — или fan-out'ит на несколько для параллельного выполнения.

Ключевые проектные решения:
- **Один интерфейс, много агентов** — пользователю не нужно знать, какой агент за какую задачу. Routing-слой Paseo решает.
- **Кросс-устройственно** — та же оркестрация работает с телефона, десктоп-приложения или CLI. Агенты крутятся серверсайд; интерфейс — тонкий клиент.
- **Параллельно по умолчанию** — когда задачи независимы, Paseo автоматически параллелит по доступным агентам, а не ставит в очередь.

### OpenClaw

**Паттерн:** нативное порождение sub-agent'ов с push-based-завершением

OpenClaw реализует multi-agent-оркестрацию как first-class-примитив harness через `sessions_spawn`. Родитель порождает sub-agent'ов одним tool-вызовом, указывая label, задачу и опционально модель/thinking-конфигурацию.

Ключевые проектные решения:
- **Push-based-завершение** — когда sub-agent заканчивает, его результат автоматически анонсируется в session родителя. Polling не нужен. Это явно задокументировано: *«Completion is push-based: it will auto-announce when done.»*
- **Отслеживание по label** — каждый sub-agent получает человекочитаемый label (например, `article-research`, `code-review`). Родитель может листать, рулить или убивать sub-agent'ов по label.
- **Ограничение глубины** — sub-agent'ам можно задать максимальную глубину, чтобы предотвратить runaway-рекурсивное порождение.
- **Инъекция context** — родитель может инъецировать конкретный context (файлы, инструкции, ограничения) в session sub-agent'а, не разделяя полную историю диалога.

## Когда НЕ использовать Multi-Agent

У multi-agent-оркестрации реальные стоимости: повышенная латентность (оверхед порождения), больший расход токенов (дублирующиеся system-промпты и context по агентам) и добавленная сложность дебага. Избегайте её, когда:

### Задача влезает в один context window

Если задача берёт 10K токенов context и 5 tool-вызовов, один агент справится быстрее и надёжнее, чем порождение воркеров. Оверхед multi-agent-координации (порождение, передача context, мерж результатов) превышает выгоду.

### Стоимость координации превышает выгоду

Задача, требующая плотного back-and-forth между двумя агентами, — где Агенту A нужен результат Агента B до следующего шага, и наоборот, — лучше обрабатывается одним агентом с последовательными tool-вызовами. Multi-agent блещет, когда работа *независима*; буксует, когда глубоко переплетена.

### Требуется строгий порядок

Если шаги должны идти в точном порядке с точными handoff'ами и параллелизма нет, один агент с пошаговым планом проще и детерминированнее. Последовательный pipeline из агентов добавляет латентность на каждой границе стадий без выигрыша в параллелизме — используйте его, только если стадии реально выигрывают от изоляции context или специализации.

## Антипаттерны

### Разделяемое mutable-состояние

Два агента, одновременно пишущие в один файл, строку БД или переменную. Это даёт race, перезаписанную работу и недетерминированные результаты. Фикс: наделите каждого агента владением над своими ресурсами. Если два агента должны вкладываться в один артефакт, используйте паттерн Fan-Out/Fan-In, где merger-агент последовательно комбинирует их выводы.

### Неограниченный Fan-Out

Порождение агентов пропорционально размеру ввода без cap'а. «Переведи этот документ на все 50 поддерживаемых языков» порождает 50 одновременных агентов и перегружает систему. Фикс: ставьте лимит `max_workers` и обрабатывайте батчами.

```python
# Bad: unbounded
for lang in all_50_languages:
    spawn_agent(task=f"Translate to {lang}")

# Good: bounded batches
BATCH_SIZE = 8
for batch in chunked(all_50_languages, BATCH_SIZE):
    results = fan_out_fan_in(
        [f"Translate to {lang}" for lang in batch],
        merge_prompt="Collect all translations.",
    )
```

### Без таймаута или circuit breaker

Sub-agent, который висит — ждёт rate-limited-API, застрял в бесконечном цикле или просто гоняет сложную задачу — блокирует всю оркестрацию, если нет таймаута. Каждое порождение sub-agent'а должно включать:
- **Таймаут**, убивающий агента после максимума длительности
- **Circuit breaker**, останавливающий retry после N неудач
- **Fallback-путь**, позволяющий родителю продолжить с частичными результатами

### Избыточная декомпозиция

Дробление 30-секундной задачи на 5 sub-agent'ов, каждый с 15 секундами оверхеда на порождение и возврат. Суммарное wall-clock растёт с 30 до 75 секунд. Multi-agent — инструмент управления сложностью и параллелизмом, а не дефолт для каждой задачи.

## Выбор правильного паттерна

| Ситуация | Паттерн | Причина |
|-----------|---------|--------|
| Стадии с разной экспертизой | Sequential Pipeline | Каждая стадия получает спец. context и tools |
| N независимых одинаковых задач | Fan-Out / Fan-In | Максимальный параллелизм, простой мерж |
| Сложная задача, требующая суждения о шагах | Supervisor | Центральное принятие решений, гибкий роутинг |
| Дебаты или многоракурсный анализ | Peer-to-Peer | Агенты представляют разные точки зрения |
| Смесь параллельной и последовательной работы | Supervisor + Fan-Out | Supervisor решает, когда параллелить |

Начните с простейшего работающего паттерна. Один агент с хорошим использованием tools справляется с 80% задач. Когда упрётесь в лимиты из [Зачем Multi-Agent?](#зачем-multi-agent), берите самый лёгкий паттерн оркестрации, решающий конкретное узкое место. Статья [Sub-Agent](sub-agent.md) разбирает простейший паттерн делегирования — начните оттуда, прежде чем строить полноценный слой оркестрации.
